Design Patterns

Gang of Four patterns frequently asked in Java Spring Boot interviews at 2–3 YOE level.

Explain the Factory Design Pattern

Creational

Factory Pattern delegates object creation to a dedicated factory class/method instead of calling new directly in client code. Client depends on an interface, not concrete classes — supports Open/Closed Principle.

// Product interface
interface Notification {
    void send(String message);
}

class EmailNotification implements Notification {
    public void send(String msg) { /* send email */ }
}
class SmsNotification implements Notification {
    public void send(String msg) { /* send SMS */ }
}

// Simple Factory
class NotificationFactory {
    static Notification create(String type) {
        return switch (type) {
            case "EMAIL" -> new EmailNotification();
            case "SMS"   -> new SmsNotification();
            default -> throw new IllegalArgumentException(type);
        };
    }
}

// Usage in Spring — often replaced by @Bean + @Qualifier or Map injection
Notification n = NotificationFactory.create("EMAIL");
n.send("Order confirmed");

Variants: Simple Factory (static method), Factory Method (subclasses decide type), Abstract Factory (families of related objects). In Spring Boot, @Configuration + @Bean methods are a built-in factory mechanism.

Explain the Observer Design Pattern

Behavioral

Observer defines a one-to-many dependency: when one object (subject) changes state, all dependent objects (observers) are notified automatically.

// Subject
interface OrderSubject {
    void subscribe(OrderObserver observer);
    void unsubscribe(OrderObserver observer);
    void notifyObservers(OrderEvent event);
}

// Observer
interface OrderObserver {
    void onOrderEvent(OrderEvent event);
}

class OrderService implements OrderSubject {
    private final List<OrderObserver> observers = new CopyOnWriteArrayList<>();

    public void placeOrder(Order order) {
        // business logic
        notifyObservers(new OrderEvent("CREATED", order));
    }
    // subscribe, unsubscribe, notifyObservers...
}

class EmailNotifier implements OrderObserver {
    public void onOrderEvent(OrderEvent e) { /* send email */ }
}

In Spring Boot: ApplicationEventPublisher + @EventListener is the framework-native Observer pattern. Kafka/RabbitMQ extend this to distributed observers across microservices.

Have you worked on Low-Level Design (LLD)?

Behavioral LLD

This is both a experience question and a design skill check. Structure your answer using a real or hypothetical example:

  • Problem: "I designed a notification module / parking lot / rate limiter for..."
  • Requirements: Functional (what it must do) + non-functional (thread safety, extensibility, scale)
  • Entities: Classes and relationships (UML or verbal: User, Order, NotificationChannel)
  • Patterns: Factory (channel creation), Strategy (payment/fee algorithm), Observer (event notification)
  • APIs: Public interfaces and method signatures
  • Concurrency: How you handle shared state (locks, immutable objects, concurrent collections)
Sample one-liner: "Yes — for our order service I designed pluggable notification channels using Strategy + Factory, with Spring's event publisher as Observer so email and SMS handlers stay decoupled from order placement logic."
