Multithreading — Beginner vs Professional

The topic that separates beginners from professional Java developers. Interviewers want when and why — not just definitions.

Why Multithreading Matters in Modern Apps

Every modern application performs multiple tasks simultaneously — HTTP requests, DB calls, file I/O, messaging. Professionals know which tool to use and why.

PROFESSIONAL MINDSET

When and why multithreading is useful — the interview answer framework

In simple words

Use threads when your app waits a lot (I/O) or must do many things at once. Don't use threads for everything — wrong threading causes bugs worse than slow code.

What to tell the interviewer
  • When: parallel API calls, background jobs, consumer queues, server handling many clients, batch processing.
  • Why: better throughput, responsive UI/API, CPU utilization while waiting on I/O.
  • When NOT: simple CRUD with no concurrency need; CPU-bound on single core without pool tuning.
  • Always tie to a project: "In our order service we used ExecutorService because…"

Beginner answer

"A thread is a lightweight process. You extend Thread class to create one."

Professional answer

"We use a fixed thread pool for async notifications so Tomcat threads aren't blocked on SMTP. Runnable tasks go to ExecutorService; shared order cache uses ConcurrentHashMap to avoid race conditions."
CONCEPT 01

Thread vs Process

In simple words

Process = separate application with its own memory. Thread = worker inside a process sharing the same heap — faster to create, but needs careful sync.

What to tell the interviewer
  • When: threads inside one Spring Boot app; separate processes for separate microservices.
  • Why threads: cheap communication via shared memory; Tomcat thread per request.
  • Why processes: isolation — payment service crash doesn't kill order service.
  • Project: one JVM runs 200 Tomcat threads; Kubernetes runs 5 pod processes behind load balancer.
CONCEPT 02

Creating Threads

In simple words

Three ways: extend Thread (avoid), implement Runnable (better), or submit to ExecutorService (production standard).

What to tell the interviewer
  • When raw Thread: almost never in enterprise — demos and interviews only.
  • When Runnable/Callable: defining task logic decoupled from thread management.
  • When ExecutorService: always in production — pool reuse, bounded concurrency, Future results.
  • Call start() not run() — run() executes in current thread.
// Production pattern
ExecutorService pool = Executors.newFixedThreadPool(8);
pool.submit(() -> generateReport(orderId));

// Spring Boot — prefer @Async + TaskExecutor bean
@Async
public void sendEmailAsync(Order order) { ... }
CONCEPT 03

Runnable Interface

In simple words

Runnable = a task with a run() method. Separates what to do from who runs it — the thread pool runs it.

What to tell the interviewer
  • Why Runnable over extending Thread: Java single inheritance; same task can run on different executors.
  • When: fire-and-forget work — logging, metrics, cache warming.
  • Need a return value? Use Callable<V> + Future instead.
  • Lambda: () -> doWork() works because Runnable is a functional interface.
Runnable task = () -> auditLog.record(event);
new Thread(task).start();           // OK for learning
executor.submit(task);              // correct for production
CONCEPT 04

Synchronization

In simple words

Only one thread at a time enters critical code that touches shared data — synchronized, locks, or concurrent collections.

What to tell the interviewer
  • When: read-modify-write on shared state (balance, inventory count, cache map).
  • Why: without sync, race conditions corrupt data under load.
  • Prefer: ConcurrentHashMap, AtomicInteger over synchronized everywhere.
  • Project: synchronized block on accountId when transferring funds; or DB row lock for inventory.
  • Keep lock scope minimal — don't hold lock during external HTTP calls.
// Too coarse — avoid holding lock during I/O
synchronized (this) {
    balance -= amount;
    callPaymentGateway(); // BAD — blocks other threads
}

// Better — lock only the state update
synchronized (this) { balance -= amount; }
callPaymentGateway();
CONCEPT 05

Race Conditions

In simple words

Two threads read the same value, both update it, one update is lost — result depends on lucky timing. Classic: count++.

What to tell the interviewer
  • Symptom: works in dev, fails randomly in prod under load.
  • Detect: stress tests, thread dumps, wrong totals in reports.
  • Fix hierarchy: immutability → thread confinement → atomic classes → locks.
  • Project story: "Ticket counter showed 998 sold but DB had 1000 — race on increment; fixed with AtomicInteger."
// Race
counter++; // not atomic

// Fixes
AtomicInteger counter = new AtomicInteger();
counter.incrementAndGet();

// or
synchronized (lock) { counter++; }
CONCEPT 06

Executor Framework

In simple words

java.util.concurrent thread pools — submit tasks instead of creating threads manually. Controls how many jobs run at once.

What to tell the interviewer
  • When: any async batch — emails, report generation, parallel external API calls.
  • Why: reuses threads (cheap), queues overflow work, returns Future for results.
  • Pool types: fixed (bounded), cached (grows/shrinks), scheduled (cron-like).
  • Production: configure core/max pool, queue size, rejection policy; monitor queue depth.
  • Spring @Async wraps ExecutorFramework for service methods.
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    4, 8, 60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(100),
    new ThreadPoolExecutor.CallerRunsPolicy()
);

Future<Report> f = executor.submit(() -> buildReport(date));
Report r = f.get(30, TimeUnit.SECONDS);

executor.shutdown();
BUILD TO LEARN

Projects that prove you understand multithreading

In simple words

The easiest way to answer confidently is building apps that actually use threads — not memorizing definitions.

What to tell the interviewer
  • Producer-Consumer: BlockingQueue + multiple producers/consumers — order pipeline.
  • Thread pool web scraper: ExecutorService fetches 50 URLs in parallel.
  • Bank transfer simulator: synchronized/locks — demonstrate race then fix.
  • Spring Boot @Async: order API returns fast, email sends in background pool.
  • Kafka consumer: multi-threaded listener container — parallel event processing.
