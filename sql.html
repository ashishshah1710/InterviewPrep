SQL and Database

Query writing, joins, aggregations, deduplication, and performance optimization techniques.

Write a query using GROUP BY and HAVING

Aggregation 
-- Find departments where average salary exceeds 50,000
-- and that have more than 5 employees
SELECT
    department_id,
    COUNT(*)          AS employee_count,
    AVG(salary)       AS avg_salary,
    MAX(salary)       AS max_salary
FROM employees
WHERE status = 'ACTIVE'          -- filter rows BEFORE grouping
GROUP BY department_id
HAVING AVG(salary) > 50000       -- filter groups AFTER aggregation
   AND COUNT(*) > 5
ORDER BY avg_salary DESC;

Key distinction: WHERE filters individual rows before grouping. HAVING filters aggregated groups after GROUP BY. You cannot use aggregate functions in WHERE.

-- UNION: returns unique cities from both tables
SELECT city FROM customers
UNION
SELECT city FROM suppliers;

-- UNION ALL: returns all cities (faster)
SELECT city FROM customers
UNION ALL
SELECT city FROM suppliers;

What is the difference between LEFT JOIN and RIGHT JOIN?

Joins

LEFT JOIN (LEFT OUTER JOIN): Returns all rows from the left table, plus matching rows from the right table. If no match exists, right-side columns are NULL.

RIGHT JOIN (RIGHT OUTER JOIN): Returns all rows from the right table, plus matching rows from the left table. If no match exists, left-side columns are NULL.

-- All employees and their departments (even if unassigned)
SELECT e.name, d.dept_name
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.id;

-- All departments and their employees (even if no employees)
SELECT e.name, d.dept_name
FROM employees e
RIGHT JOIN departments d ON e.dept_id = d.id;

Practical note: A RIGHT JOIN B is equivalent to B LEFT JOIN A. LEFT JOIN is preferred in practice for readability — always put the "primary" table on the left.

How would you retrieve the second-highest salary from a table?

Subquery Window Functions 
-- Approach 1: Subquery (works everywhere)
SELECT MAX(salary) AS second_highest
FROM employees
WHERE salary < (SELECT MAX(salary) FROM employees);

-- Approach 2: DENSE_RANK (handles ties correctly)
SELECT salary
FROM (
    SELECT salary,
           DENSE_RANK() OVER (ORDER BY salary DESC) AS rnk
    FROM employees
) ranked
WHERE rnk = 2;

-- Approach 3: LIMIT/OFFSET (PostgreSQL, MySQL)
SELECT DISTINCT salary
FROM employees
ORDER BY salary DESC
LIMIT 1 OFFSET 1;

-- Approach 4: Self-join count
SELECT DISTINCT e1.salary
FROM employees e1
WHERE 1 = (
    SELECT COUNT(DISTINCT e2.salary)
    FROM employees e2
    WHERE e2.salary > e1.salary
);

Interview tip: Clarify whether "second highest" means exactly one value or should handle ties. DENSE_RANK is the most robust approach for tie handling.

How do you eliminate duplicate records in SQL?

Deduplication 
-- Method 1: DELETE keeping the lowest ID (PostgreSQL)
DELETE FROM employees a
USING employees b
WHERE a.id > b.id
  AND a.email = b.email;

-- Method 2: ROW_NUMBER window function (most flexible)
DELETE FROM employees
WHERE id IN (
    SELECT id FROM (
        SELECT id,
               ROW_NUMBER() OVER (
                   PARTITION BY email ORDER BY created_at ASC
               ) AS rn
        FROM employees
    ) t
    WHERE rn > 1
);

-- Method 3: Prevention — add UNIQUE constraint
ALTER TABLE employees ADD CONSTRAINT uq_email UNIQUE (email);

Always prefer prevention (unique constraints) over cleanup. Use ROW_NUMBER when you need control over which duplicate to keep (e.g., keep the oldest record).

How do you optimize a slow-performing SQL query?

Performance
  • EXPLAIN / EXPLAIN ANALYZE: Always start here. Identify full table scans, missing indexes, and expensive sort/hash operations.
  • Add indexes: On columns in WHERE, JOIN, ORDER BY, and GROUP BY clauses. Use composite indexes for multi-column filters (leftmost prefix rule).
  • Avoid SELECT *: Fetch only needed columns to reduce I/O and enable index-only scans.
  • Rewrite subqueries as JOINs: Correlated subqueries execute per row; JOINs are often more efficient.
  • Use pagination wisely: Replace OFFSET with keyset/cursor pagination for large tables (WHERE id > last_seen_id LIMIT 20).
  • Denormalize or materialize: Pre-compute heavy aggregations in materialized views for reporting queries.
  • Update statistics: Run ANALYZE so the query planner has accurate row count estimates.
  • Partition tables: Split large tables by date or range so queries scan fewer partitions.
  • Connection pooling: Use PgBouncer/HikariCP to avoid connection overhead.

Find the highest salary for each department

GROUP BY Window Functions 
-- Approach 1: GROUP BY + MAX (most common interview answer)
SELECT department_id, MAX(salary) AS highest_salary
FROM employees
GROUP BY department_id;

-- Approach 2: With department name (JOIN)
SELECT d.dept_name, MAX(e.salary) AS highest_salary
FROM employees e
JOIN departments d ON e.department_id = d.id
GROUP BY d.dept_name;

-- Approach 3: Employee names who earn the max (handle ties)
SELECT department_id, name, salary
FROM (
    SELECT department_id, name, salary,
           RANK() OVER (PARTITION BY department_id ORDER BY salary DESC) AS rnk
    FROM employees
) ranked
WHERE rnk = 1;

Follow-up: If two employees tie for highest salary, MAX + GROUP BY still returns one max value; use RANK() or DENSE_RANK() to return all top earners.
