TOPIC 01 · JAVA STREAMS

Find the length of each element using Streams

In simple words

Use stream map to turn each string into its length — one line per word, no manual for-loop.

What to tell the interviewer
  • Say: map each element to length with String::length or s -> s.length().
  • Show collect to List or print with forEach.
  • Mention time O(n), space O(n) for result list.
List<String> words = List.of("java", "spring", "boot");

// Length of each element
List<Integer> lengths = words.stream()
    .map(String::length)
    .toList();
// [4, 6, 4]

words.stream()
    .mapToInt(String::length)
    .forEach(System.out::println);

// With index if needed
IntStream.range(0, words.size())
    .forEach(i -> System.out.println(words.get(i) + " -> " + words.get(i).length()));
TOPIC 02 · HASHMAP BEHAVIOUR

What will be the output? put("abc","tan"); put("abc","cho"); get("abc");

In simple words

HashMap does not allow duplicate keys. The second put replaces the first value for the same key.

What to tell the interviewer
  • Answer: "cho" — second put overwrites "tan".
  • HashMap stores unique keys; put on existing key updates value only.
  • Size stays 1 — only one entry for key "abc".
  • Contrast: if you need multiple values per key, use Map<K, List<V>> or Multimap.
Map<String, String> map = new HashMap<>();
map.put("abc", "tan");
map.put("abc", "cho");
System.out.println(map.get("abc")); // "cho"
System.out.println(map.size());   // 1
TOPIC 03 · MISSING NUMBER

Find the missing number from int[] arr = {1, 2, 5, 6, 3, 7}

In simple words

Missing number is 4. For huge arrays, use math (expected sum minus actual sum) or XOR — no need to sort or scan twice.

What to tell the interviewer
  • Small array: sort and scan, or use a Set — O(n log n) or O(n).
  • Large array: expected sum = n×(n+1)/2 minus actual sum → missing number in O(n) time, O(1) space.
  • Alternative: XOR all indices 1..n XOR all array values → result is missing number.
  • Watch overflow on sum — use long for very large n.
// Sum formula — O(n) time, O(1) space
public static int missingNumber(int[] arr, int n) {
    long expected = (long) n * (n + 1) / 2;
    long actual = 0;
    for (int x : arr) actual += x;
    return (int) (expected - actual);
}
// arr has n-1 elements from 1..n → missing = 4

// XOR — no overflow risk
public static int missingXor(int[] arr, int n) {
    int xor = 0;
    for (int i = 1; i <= n; i++) xor ^= i;
    for (int x : arr) xor ^= x;
    return xor;
}
TOPIC 04 · SPRING ANNOTATIONS

Explain @Autowired, @PathVariable, and @RequestParam

In simple words

@Autowired = Spring injects a dependency. @PathVariable = value from URL path. @RequestParam = value from query string (?key=value).

What to tell the interviewer
  • @Autowired: DI by type — constructor injection preferred in production.
  • @PathVariable: part of URI template — /orders/{id} → id=101.
  • @RequestParam: query param — /orders?status=ACTIVE.
  • Show one controller method using all three concepts.
@RestController
@RequestMapping("/api/orders")
public class OrderController {

    private final OrderService orderService; // @Autowired via constructor

    public OrderController(OrderService orderService) {
        this.orderService = orderService;
    }

    @GetMapping("/{id}")
    public Order getOrder(
            @PathVariable Long id,
            @RequestParam(defaultValue = "false") boolean includeItems) {
        return orderService.findById(id, includeItems);
    }
}
TOPIC 05 · SORT MAP

Sort a Map<String, Integer>

In simple words

Convert map entries to a stream, sort by key or value, collect into LinkedHashMap (order preserved) or TreeMap (auto-sorted).

What to tell the interviewer
  • By value desc: stream entries → sorted by Map.Entry.comparingByValue(Comparator.reverseOrder()).
  • Collect to LinkedHashMap to keep sort order.
  • Or use TreeMap with custom comparator for always-sorted map.
Map<String, Integer> map = new HashMap<>();
map.put("apple", 3);
map.put("banana", 1);
map.put("cherry", 2);

// Sort by value ascending
Map<String, Integer> byValue = map.entrySet().stream()
    .sorted(Map.Entry.comparingByValue())
    .collect(Collectors.toMap(
        Map.Entry::getKey,
        Map.Entry::getValue,
        (a, b) -> a,
        LinkedHashMap::new
    ));

// Sort by key — TreeMap
Map<String, Integer> byKey = new TreeMap<>(map);
TOPIC 06 · OOP

Difference between Encapsulation and Abstraction

In simple words

Encapsulation = hide data inside a class (private fields + getters). Abstraction = hide complexity — show only what user needs (interface or abstract idea).

What to tell the interviewer
  • Encapsulation: data hiding — private fields, controlled access via methods.
  • Abstraction: design-level hiding — PaymentService interface, user doesn't see DB logic.
  • Encapsulation protects object state; abstraction simplifies usage.
  • Example: ATM button (abstraction) vs account balance stored privately (encapsulation).

Hides data (implementation detail of one class)Hides complexity (what object does, not how)

TOPIC 07 · CUSTOM EXCEPTION

How do you create a Custom Exception in Java?

In simple words

Extend RuntimeException (unchecked, common in Spring) or Exception (checked). Add constructors that pass message to super.

What to tell the interviewer
  • Extend RuntimeException for business errors mapped to HTTP 4xx/5xx.
  • Include meaningful message and optional error code field.
  • Handle globally with @ControllerAdvice — don't catch in every method.
public class InsufficientBalanceException extends RuntimeException {
    private final String accountId;

    public InsufficientBalanceException(String accountId, BigDecimal required) {
        super("Insufficient balance for account " + accountId + ", required: " + required);
        this.accountId = accountId;
    }
}

// Usage
throw new InsufficientBalanceException(accountId, amount);
TOPIC 08 · CUSTOM CLASS

How do you create a Custom Class in Java?

In simple words

Define a class with fields, constructor to set them, getters (and setters if mutable). Override equals/hashCode if used in collections.

What to tell the interviewer
  • Fields, constructor, accessors — basic POJO.
  • Override equals/hashCode if object goes in HashMap/HashSet.
  • Consider immutable design (final fields) for thread safety.
  • Java 16+ record for simple data carriers.
public class Employee {
    private final Long id;
    private String name;
    private double salary;

    public Employee(Long id, String name, double salary) {
        this.id = id;
        this.name = name;
        this.salary = salary;
    }

    public Long getId() { return id; }
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
    public double getSalary() { return salary; }
}
TOPIC 09 · MAP COMPARISON

Difference between HashMap, Hashtable, and ConcurrentHashMap

In simple words

HashMap = fast, not thread-safe, allows null. Hashtable = old, synchronized on every call, slow, no null. ConcurrentHashMap = modern, thread-safe, fine-grained locking.

What to tell the interviewer
  • Hashtable is legacy — don't use in new code.
  • HashMap for single-threaded; CHM for concurrent apps.
  • CHM: no null keys/values; better throughput than Hashtable.

TOPIC 10 · CONCURRENCY CAPACITY

How many elements can ConcurrentHashMap process at a time? What about HashMap?

In simple words

There is no fixed limit on how many entries you can store — both grow as needed. The real question is how many threads can read/write safely at once.

What to tell the interviewer
  • Clarify: they likely mean concurrent access, not element count.
  • HashMap: many threads = unsafe — can corrupt data; use external sync.
  • CHM: multiple threads can read concurrently; writes lock only one bucket.
  • Default concurrency level (legacy) was 16 segments — Java 8+ uses bucket-level locking, scales with cores and bucket count.
  • Element capacity: both limited by heap memory, not a fixed thread-style cap.
Trap answer to avoid: Don't say "16 elements" — say unlimited entries (memory-bound), but concurrent threads are handled safely by CHM without locking the entire map.
TOPIC 11 · REST HTTP

Difference between PUT and PATCH

In simple words

PUT replaces the entire resource. PATCH updates only the fields you send — partial update.

What to tell the interviewer
  • PUT: send full object — missing fields may become null/default.
  • PATCH: send only changed fields — e.g. update email only.
  • PUT is idempotent; PATCH should be designed idempotent too.
  • Spring: @PutMapping vs @PatchMapping with JsonMergePatch or custom DTO.
// PUT /users/1 — replace entire user
{ "name": "Alice", "email": "a@x.com", "role": "ADMIN" }

// PATCH /users/1 — update email only
{ "email": "new@x.com" }
TOPIC 12 · SQL DDL

Difference between DROP and TRUNCATE

In simple words

DROP deletes the table structure and all data — table gone. TRUNCATE deletes all rows but keeps the empty table structure — faster, cannot rollback easily in some DBs.

What to tell the interviewer
  • DROP TABLE: removes metadata + data; need CREATE to use again.
  • TRUNCATE: deallocates data pages; structure, indexes remain.
  • TRUNCATE faster than DELETE without WHERE; minimal logging.
  • TRUNCATE fails if FK references exist; DELETE can row-by-row with constraints.
DROPTRUNCATEDELETE
Removes table objectEmpties tableRemoves rows (with WHERE)

TOPIC 13 · IMMUTABLE CLASS

How do you create an Immutable Class in Java?

In simple words

Final class, all fields final and private, no setters, copy mutable objects in constructor, return unmodifiable views from getters.

What to tell the interviewer
  • 1) final class 2) private final fields 3) constructor-only init
  • Defensive copy: List.copyOf() in constructor and getters
  • Thread-safe without locks — good for DTOs shared across threads
  • Java Records are immutable by default for simple cases
public final class Money {
    private final BigDecimal amount;
    private final String currency;

    public Money(BigDecimal amount, String currency) {
        this.amount = amount;
        this.currency = currency;
    }
    public BigDecimal getAmount() { return amount; }
    public String getCurrency() { return currency; }
    // no setters
}
TOPIC 14 · STREAMS

Difference between map() and flatMap()

In simple words

map = 1 input → 1 output. flatMap = 1 input → many outputs, then flatten into one stream.

What to tell the interviewer
  • map: List<Order> → List<List<Item>> (nested)
  • flatMap: List<Order> → List<Item> (flat)
  • Also used with Optional to chain without nested if-present checks
// map — one-to-one
List<String> upper = words.stream().map(String::toUpperCase).toList();

// flatMap — one-to-many, flattened
List<String> allWords = sentences.stream()
    .flatMap(s -> Arrays.stream(s.split(" ")))
    .toList();
TOPIC 15 · SERIALIZATION

What is Serialization and why is it used?

In simple words

Turn a Java object into bytes to save or send over network. Deserialization rebuilds the object from bytes.

What to tell the interviewer
  • Implements Serializable — object → byte stream → file/socket/cache
  • Uses: RMI (legacy), session replication, Kafka (prefer JSON/Avro today)
  • serialVersionUID for version compatibility
  • Security risk: never deserialize untrusted data — use JSON + validation instead in modern apps
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    private String name;
    private transient String password; // not serialized
}
TOPIC 16 · EXCEPTION HANDLING

How do you handle Exceptions in Java?

In simple words

try/catch/finally catches errors; throw sends error up; custom exceptions for business rules; global handler in Spring for APIs.

What to tell the interviewer
  • Checked: must handle (IOException). Unchecked: RuntimeException.
  • Don't swallow exceptions — log with context
  • Spring: @ControllerAdvice maps exceptions to HTTP status + JSON body
  • Fail fast on unrecoverable errors; retry only idempotent operations
try {
    processPayment(order);
} catch (PaymentFailedException e) {
    log.error("Payment failed for order {}", order.getId(), e);
    throw e; // or return fallback
} finally {
    MDC.clear();
}

TOPIC 18 · PRODUCTION SUPPORT

If a production issue occurs, how would you investigate after checking the logs?

In simple words

After logs: check metrics/dashboards, reproduce scope, thread dump or heap if JVM issue, DB slow queries, recent deploys, then fix and document RCA.

What to tell the interviewer
  • 1) Confirm impact — how many users, which API, since when
  • 2) Metrics: CPU, memory, error rate, latency (Grafana/Dynatrace)
  • 3) Correlate with deploy/config change
  • 4) JVM: jstack (threads), heap dump (OOM), GC logs
  • 5) DB: slow query log, connection pool exhaustion
  • 6) External deps: downstream timeout, circuit breaker open
  • 7) Mitigate (rollback/scale), then RCA and preventive action
Logs → Metrics → Recent change? → Thread/Heap dump → DB/Network → Mitigate → Root cause → Fix + alert + runbook update
