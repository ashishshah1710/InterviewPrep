20 Must-Know Concurrency Concepts

Core concurrency building blocks + deep-dive FAQ answers for senior Java interviews.

20 Must-Know Concurrency Concepts

From concurrency vs parallelism to blocking queues — each concept in its own block with simple explanation and interview script.

Core concepts 1 – 20

CONCEPT 01

Concurrency — Tasks interleaving execution on a single core

In simple words

One CPU switches fast between tasks so they appear to run together — like one chef chopping while soup simmers, switching back and forth.

What to tell the interviewer
  • Concurrency = dealing with many things at once (structure), not necessarily at the same instant.
  • OS time-slices threads on one core — interleaved execution.
  • I/O-bound apps benefit: while one thread waits on DB, another runs.
CONCEPT 02

Parallelism — Tasks executing simultaneously on multiple cores

In simple words

Multiple cores truly run work at the same time — like four chefs each cooking a different dish simultaneously.

What to tell the interviewer
  • Parallelism = simultaneous execution — needs multiple CPU cores.
  • Java: parallelStream(), ForkJoinPool, multiple threads on multi-core machine.
  • You can have concurrency without parallelism (1 core); parallelism usually includes concurrency.
CONCEPT 03

Processes vs Threads — Isolated buckets vs lightweight units

In simple words

Process = separate app with its own memory. Thread = worker inside a process sharing the same memory heap.

What to tell the interviewer
  • Process isolation: crash in one JVM app doesn't kill another process.
  • Threads share heap — fast communication but need synchronization.
  • Creating a process is heavy; creating a thread is lighter.
CONCEPT 04

Thread Lifecycle — New → Runnable → Running → Blocked → Terminated

In simple words

A thread is born, waits its turn, runs, pauses when waiting for lock/I/O, then dies when done.

What to tell the interviewer
  • NEW → start() → RUNNABLE → scheduler picks → RUNNING.
  • BLOCKED/WAITING/TIMED_WAITING: waiting for lock, wait(), sleep(), join().
  • TERMINATED: run() completes or uncaught exception.
  • jstack shows thread states during production incidents.
NEW → start() → RUNNABLE ⇄ RUNNING ↓ blocked/wait/sleep/I/O BLOCKED / WAITING / TIMED_WAITING ↓ run completes TERMINATED
CONCEPT 05

Race Conditions — Output depends on unpredictable timing

In simple words

Two threads update the same variable without coordination — who wins depends on luck, so results are wrong sometimes.

What to tell the interviewer
  • Classic: count++ — read-modify-write, not atomic.
  • Symptoms: flaky tests, bugs only under production load.
  • Fix: locks, atomic variables, immutable data, or thread confinement.
// Race: both read count=5, both write 6
private int count = 0;
count++; // NOT safe without sync or AtomicInteger
CONCEPT 06

Mutex — Mutual exclusion lock for exclusive single-thread access

In simple words

Only one thread at a time can enter the protected code — like one person in a bathroom with a lock on the door.

What to tell the interviewer
  • Java: synchronized block/method, ReentrantLock.
  • Protects critical section — shared counter, balance update.
  • Trade-off: correctness vs potential contention and deadlocks.
CONCEPT 07

Semaphores — Signaling counter controlling access for N threads

In simple words

A permit counter — N threads can enter at once; others wait until a permit is released. Like N parking slots.

What to tell the interviewer
  • Java: Semaphore(N) — acquire() takes permit, release() returns it.
  • Limit concurrent DB connections or API calls to N.
  • Binary semaphore (1 permit) behaves like mutex but different semantics (see FAQ).
Semaphore sem = new Semaphore(5); // max 5 concurrent
sem.acquire();
try { callExternalApi(); }
finally { sem.release(); }
CONCEPT 08

Condition Variables — Safe queues letting threads sleep until signaled

In simple words

Threads wait on a condition (queue empty/full) and get woken when another thread signals — no busy-waiting.

What to tell the interviewer
  • Java: Lock + Condition (await/signal) or wait/notify on monitor.
  • Producer waits when buffer full; consumer waits when empty.
  • Always use while loop to re-check condition (spurious wakeup).
CONCEPT 09

Coarse vs Fine Locking — Big slow lock vs small fast locks

In simple words

Coarse = one lock for everything (simple but slow). Fine = many small locks (complex but less contention).

What to tell the interviewer
  • Coarse: synchronized entire map — one writer blocks all readers.
  • Fine: ConcurrentHashMap bucket locks — threads hit different buckets.
  • Start coarse; optimize to fine when profiling shows lock contention.
CONCEPT 10

Reentrant Locks — Thread can re-acquire its own lock

In simple words

Same thread can lock twice without deadlocking itself — Java synchronized and ReentrantLock are both reentrant.

What to tell the interviewer
  • Thread holds count: lock → call method that locks again → count=2 → unlock twice.
  • Needed when synchronized method calls another synchronized method on same object.
  • ReentrantLock also reentrant by name and design.
CONCEPT 11

Try-Lock — Non-blocking attempt that returns immediately if busy

In simple words

Try to get the lock — if someone else has it, don't wait forever; return false or use fallback.

What to tell the interviewer
  • lock.tryLock() or tryLock(timeout, unit) — avoids indefinite blocking.
  • Use to prevent deadlock: try locks in order with timeout, back off if fail.
  • Good for optional work: skip cache refresh if lock busy.
if (lock.tryLock(500, TimeUnit.MILLISECONDS)) {
    try { updateCache(); }
    finally { lock.unlock(); }
} else { return staleData(); }
CONCEPT 12

CAS (Compare-and-Swap) — Atomic CPU instruction for lock-free updates

In simple words

CPU atomically says: if memory equals expected, swap to new value — else retry. No lock needed for simple updates.

What to tell the interviewer
  • Used by AtomicInteger, ConcurrentHashMap internals.
  • Lock-free but may retry under contention (optimistic).
  • See FAQ below for ABA problem.
CONCEPT 13

Deadlock — Circular wait where everyone is stuck

In simple words

Thread A holds lock 1, wants lock 2. Thread B holds lock 2, wants lock 1. Neither can proceed.

What to tell the interviewer
  • Detect with jstack — "Found one Java-level deadlock".
  • Prevent: lock ordering, timeouts, tryLock.
  • See FAQ: four Coffman conditions.
CONCEPT 14

Livelock — Active responses but zero progress

In simple words

Threads aren't blocked but keep yielding to each other — like two people in a hallway both stepping aside the same way forever.

What to tell the interviewer
  • Differs from deadlock: threads are RUNNABLE, CPU busy, no forward progress.
  • Fix: random backoff, priority, or assign roles (leader election).
  • Example: tryLock loops retrying instantly without backoff.
CONCEPT 15

Signaling Pattern — One thread explicitly waking another

In simple words

Worker waits until boss calls notify/signal — "data is ready, you can proceed."

What to tell the interviewer
  • wait/notify, Condition.await/signal, CountDownLatch.countDown().
  • Always signal under same lock that wait uses.
  • Prefer higher-level constructs (BlockingQueue, Latch) over raw wait/notify.
CONCEPT 16

Thread Pool — Reusing fixed threads to avoid allocation overhead

In simple words

Keep N workers alive; submit tasks to a queue instead of creating new Thread for every request.

What to tell the interviewer
  • ExecutorService / ThreadPoolExecutor in Java.
  • Controls max concurrency — protects CPU and downstream systems.
  • Spring @Async uses TaskExecutor thread pool.
CONCEPT 17

Producer-Consumer — Decouple generation from processing via buffer

In simple words

Producers fill a queue; consumers drain it — each side works at its own pace.

What to tell the interviewer
  • BlockingQueue is the standard Java implementation.
  • Used in log pipelines, order events, thread pool work queue.
  • Backpressure: full queue blocks producers.
CONCEPT 18

Reader-Writer — Many readers OR one writer

In simple words

Many threads can read together, but writing requires exclusive access — good when reads dominate.

What to tell the interviewer
  • Java: ReentrantReadWriteLock — readLock() shared, writeLock() exclusive.
  • Config caches, reference data — frequent reads, rare updates.
  • See FAQ: when vs standard mutex.
CONCEPT 19

Thread-Safe Cache — Safe concurrent in-memory store

In simple words

Multiple threads read/write cache without corrupting data — use ConcurrentHashMap + computeIfAbsent or Caffeine cache.

What to tell the interviewer
  • ConcurrentHashMap.computeIfAbsent prevents duplicate loads.
  • Caffeine/Guava Cache handles eviction and concurrency.
  • Define TTL and invalidation on writes for consistency.
cache.computeIfAbsent(key, k -> loadFromDb(k));
CONCEPT 20

Blocking Queue — Blocks when full (producer) or empty (consumer)

In simple words

put() waits if queue full; take() waits if empty — built-in producer-consumer without manual wait/notify.

What to tell the interviewer
  • ArrayBlockingQueue (bounded), LinkedBlockingQueue, PriorityBlockingQueue.
  • ThreadPoolExecutor internal work queue is a BlockingQueue.
  • See FAQ: implementing bounded queue with Condition variables.

FAQ · CAS

How does Compare-and-Swap work under the hood, and what is the ABA problem?

In simple words

CAS atomically checks "is memory still X?" — if yes, set to Y; if no, fail and retry. ABA = value looks same (A) but changed A→B→A in between, fooling some algorithms.

What to tell the interviewer
  • CAS loop: read current → compute new → compareAndSet(expected, new) → retry if fail.
  • Hardware (cmpxchg on x86) guarantees atomicity of compare+swap.
  • ABA problem: pointer/object recycled — AtomicStampedReference adds version stamp.
  • Java Atomic* classes hide CAS; ConcurrentHashMap uses CAS for some updates.
// CAS logic (conceptual)
do {
    expected = current;
    newVal = expected + 1;
} while (!compareAndSet(expected, newVal));

FAQ · DEADLOCK

What are the four Coffman conditions required for a deadlock?

In simple words

All four must be true for deadlock. Break any one condition and deadlock cannot happen.

What to tell the interviewer
  • 1) Mutual exclusion — resource held by only one thread at a time.
  • 2) Hold and wait — thread holds one resource while waiting for another.
  • 3) No preemption — resources can't be forcibly taken away.
  • 4) Circular wait — cycle in wait graph (A→B→C→A).
  • Prevention: lock ordering breaks circular wait; tryLock breaks hold-and-wait.
FAQ · MUTEX VS SEMAPHORE

What is the core difference between a Mutex and a Binary Semaphore?

In simple words

Both allow one thread at a time, but mutex = ownership (only locker unlocks). Binary semaphore = signaling (any thread can signal/release).

What to tell the interviewer
  • Mutex: lock/unlock must be same thread — protects critical section.
  • Binary semaphore: thread A can acquire, thread B can release — signaling pattern.
  • Mutex solves mutual exclusion; semaphore solves resource counting + signaling.
  • Java ReentrantLock ≈ mutex; Semaphore(1) ≈ binary semaphore but different semantics.
MutexBinary Semaphore
Ownership — only owner unlocksNo ownership — any thread may release
Protect shared dataSignal event / permit availability
FAQ · DESIGN PATTERN

How do you implement a bounded Blocking Queue using Condition Variables?

In simple words

Use one lock, two conditions: notFull (producer waits) and notEmpty (consumer waits). Array as ring buffer with head/tail.

What to tell the interviewer
  • ReentrantLock + two Condition objects: notFull, notEmpty.
  • put: while full → notFull.await(); add item; signal notEmpty.
  • take: while empty → notEmpty.await(); remove item; signal notFull.
  • Use while not if — spurious wakeup and multiple waiters.
class BoundedQueue<T> {
    final Lock lock = new ReentrantLock();
    final Condition notFull = lock.newCondition();
    final Condition notEmpty = lock.newCondition();
    final Object[] items;
    int head, tail, count;

    void put(T item) throws InterruptedException {
        lock.lock();
        try {
            while (count == items.length) notFull.await();
            items[tail] = item;
            tail = (tail + 1) % items.length;
            count++;
            notEmpty.signal();
        } finally { lock.unlock(); }
    }

    T take() throws InterruptedException {
        lock.lock();
        try {
            while (count == 0) notEmpty.await();
            T item = (T) items[head];
            head = (head + 1) % items.length;
            count--;
            notFull.signal();
            return item;
        } finally { lock.unlock(); }
    }
}
FAQ · PERFORMANCE

When do you prefer a Read-Write Lock over a standard Mutex?

In simple words

When reads are many and writes are rare — multiple readers run together; writers still get exclusive access.

What to tell the interviewer
  • Prefer RW lock: read-heavy cache, config, reference data (90%+ reads).
  • Standard mutex if writes are frequent — RW lock overhead not worth it.
  • Watch writer starvation — some implementations aren't fair.
  • Alternative: immutable snapshots + atomic reference — no locks on read path.
Use ReadWriteLockUse simple Mutex
Read ratio very highBalanced or write-heavy workload
Read operations take timeCritical section is very short
Shared config / cacheCounter updates, payment balance
