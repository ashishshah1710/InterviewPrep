Microservices and DevOps

Architecture decisions, messaging systems, CI/CD pipelines, containerization, and distributed system patterns.

Why was a microservices architecture chosen over a monolithic architecture?

Architecture

Key reasons to choose microservices:

  • Independent deployability: Deploy the payment service without redeploying the entire application.
  • Technology flexibility: Use Python for ML services, Java for transactional services, Node.js for real-time features.
  • Scalability: Scale only the bottleneck service (e.g., 10 instances of search, 2 of billing).
  • Team autonomy: Small teams own a service end-to-end — development, deployment, and operations.
  • Fault isolation: A crash in the notification service doesn't bring down order processing.
  • Faster release cycles: Smaller codebases mean faster builds, tests, and deployments.

Trade-offs: Distributed complexity (network latency, partial failures), data consistency challenges, increased operational overhead (monitoring, tracing, service discovery), and higher initial infrastructure cost.

How does Kafka differ from RabbitMQ, and when would you choose one over the other?

Messaging
FeatureApache KafkaRabbitMQ
ArchitectureDistributed commit log / event streamTraditional AMQP message broker
Message retentionPersistent, configurable retention, replayableConsumed messages typically deleted
ThroughputMillions of messages/secTens of thousands/sec
OrderingGuaranteed per partitionPer queue
RoutingTopic + partition modelExchanges, bindings, routing keys (flexible)
Consumer modelPull-based (consumer polls)Push-based (broker delivers)
Message replayYes — reset offset and re-readNo — once consumed, message is gone

Choose Kafka when: Event sourcing, log aggregation, stream processing (Kafka Streams/Flink), high-volume analytics, need to replay events, multiple consumers reading the same stream independently.

Choose RabbitMQ when: Task/job queues, RPC-style request-reply, complex routing (topic/direct/fanout exchanges), low-latency messaging, work queue patterns with acknowledgments.

What are the key features of Istio and Service Mesh in Kubernetes?

Service Mesh Kubernetes

A service mesh is an infrastructure layer that handles service-to-service communication. Istio is the most popular implementation, using an Envoy sidecar proxy injected alongside each application pod.

  • Traffic management: Canary deployments, A/B testing, traffic splitting, retries, timeouts, circuit breaking — all without changing application code.
  • Security (mTLS): Automatic mutual TLS encryption between all services. Certificate management and rotation handled by Istio.
  • Observability: Automatic distributed tracing (Jaeger/Zipkin), metrics (Prometheus), and access logs for every request — without instrumentation in app code.
  • Policy enforcement: Rate limiting, access control (RBAC), quota management at the mesh level.
  • Resilience: Outlier detection (eject unhealthy hosts), connection pooling, fault injection for chaos testing.

How do you Dockerize a Spring Boot application?

Docker 
# Multi-stage Dockerfile
FROM maven:3.9-eclipse-temurin-17 AS build
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline
COPY src ./src
RUN mvn clean package -DskipTests

FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
COPY --from=build /app/target/*.jar app.jar
USER appuser
EXPOSE 8080
HEALTHCHECK --interval=30s --timeout=3s \
  CMD wget -qO- http://localhost:8080/actuator/health || exit 1
ENTRYPOINT ["java", "-jar", "app.jar"]

Best practices: Multi-stage build (smaller final image), use JRE not JDK in runtime, run as non-root user, add health checks, use .dockerignore, leverage Spring Boot's spring-boot:build-image with Cloud Native Buildpacks for optimized images without writing a Dockerfile.

How do you manage distributed transactions across microservices?

Distributed Transactions

1. Saga Pattern (most common): A series of local transactions, each with a compensating action on failure.

  • Choreography: Services publish events; other services react. Decentralized, but harder to track.
  • Orchestration: A central saga orchestrator coordinates steps and compensations. Easier to monitor.

2. Outbox Pattern: Write business data and an event to the same local DB in one transaction. A separate process reads the outbox table and publishes to a message broker — ensures at-least-once delivery.

3. Two-Phase Commit (2PC): Coordinator asks all participants to prepare, then commit. Strong consistency but slow, tight coupling, and single point of failure. Rarely used in microservices.

4. Idempotency: Design all operations to be safely retried using idempotency keys — essential for reliable sagas.

What is the Circuit Breaker pattern, and when should it be implemented?

Resilience

The Circuit Breaker prevents cascading failures by stopping calls to a failing downstream service, giving it time to recover.

┌──────────┐ Calls ──│ CLOSED │── Normal operation, requests pass through └────┬─────┘ │ Failure threshold exceeded ▼ ┌──────────┐ │ OPEN │── Fail fast, return fallback immediately └────┬─────┘ │ After timeout period ▼ ┌──────────┐ │HALF-OPEN │── Allow one test request └────┬─────┘ Success → CLOSED | Failure → OPEN

Implement when: Calling external/unreliable services, experiencing frequent timeouts, need graceful degradation (return cached/default response), preventing thread pool exhaustion from slow downstream calls.

Libraries: Resilience4j (recommended for Spring Boot 3), Spring Cloud Circuit Breaker, Istio outlier detection at the mesh level.
