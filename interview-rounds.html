Difference between HashMap and ConcurrentHashMap

Core Java Round 1

HashMap is not thread-safe — concurrent writes can corrupt internal structure or cause infinite loops (Java 7). ConcurrentHashMap allows concurrent reads and controlled concurrent writes using bucket-level locking and CAS (Java 8+). No null keys/values in CHM. Iterator is weakly consistent, not fail-fast.

Explain Java Memory Model

JVM Round 1

JVM memory divides into: Stack (per-thread — method frames, local vars), Heap (shared — all objects; Young Gen + Old Gen, GC managed), Metaspace (class metadata, static fields). Object references live on stack; actual objects live on heap.

JMM (Java Memory Model) additionally defines visibility rules between threads — without volatile or synchronization, one thread's writes may not be visible to another due to CPU cache reordering.

Explain Exception Handling and Custom Exceptions

Core Java Round 1

Java exceptions: Checked (must handle — IOException) vs Unchecked (RuntimeException — NullPointerException). Flow: try → exception thrown → matching catch → finally always runs.

// Custom unchecked exception — preferred in Spring Boot services
public class OrderNotFoundException extends RuntimeException {
    public OrderNotFoundException(Long orderId) {
        super("Order not found: " + orderId);
    }
}

@Service
public class OrderService {
    public Order getOrder(Long id) {
        return repo.findById(id)
            .orElseThrow(() -> new OrderNotFoundException(id));
    }
}
// Global handler maps it to HTTP 404 — see @ControllerAdvice

Best practices: Don't catch generic Exception unless at top level; never swallow exceptions; use custom exceptions for domain errors; log with context; prefer unchecked in service layer (Spring translates to HTTP status).

What is an Immutable Class and how would you create one?

OOP Round 1
  • Declare class final — prevent subclassing
  • All fields private final
  • No setters — state set only via constructor
  • Defensive copy mutable inputs (List, Date) in constructor
  • Return unmodifiable copies from getters — List.copyOf(), Collections.unmodifiableList()
public final class Employee {
    private final String name;
    private final List<String> skills;

    public Employee(String name, List<String> skills) {
        this.name = name;
        this.skills = List.copyOf(skills); // defensive copy
    }
    public String getName() { return name; }
    public List<String> getSkills() { return skills; } // already immutable
}

Benefits: thread-safe without locks, safe to share, usable as HashMap keys, easier reasoning.

How does @Transactional work internally?

Spring Boot Round 1

Spring creates an AOP proxy (JDK or CGLIB) around your service bean. When a @Transactional method is called:

  • Proxy opens a DB transaction (via PlatformTransactionManager)
  • Calls your actual method
  • On success → commit; on RuntimeException → rollback (default)
  • Checked exceptions do NOT rollback unless rollbackFor specified
Common pitfall: @Transactional on private methods or self-invocation (this.save() from same class) — proxy is bypassed, no transaction! Call through injected bean or refactor. 
@Transactional(rollbackFor = Exception.class)
public void placeOrder(Order order) {
    orderRepo.save(order);       // same transaction
    inventoryRepo.decrement();   // rolls back together on failure
}

Find the first non-repeating character in a String

Coding Round 1 
public static Character firstNonRepeating(String s) {
    Map<Character, Integer> freq = new LinkedHashMap<>();
    for (char c : s.toCharArray()) {
        freq.merge(c, 1, Integer::sum);
    }
    for (Map.Entry<Character, Integer> e : freq.entrySet()) {
        if (e.getValue() == 1) return e.getKey();
    }
    return null;
}
// "aabbcdd" → 'c'   |   Time O(n), Space O(1) — max 26/128 chars

Find duplicate elements in an array

Coding Round 1 
// Approach 1: HashSet — O(n) time, O(n) space
public static Set<Integer> findDuplicates(int[] arr) {
    Set<Integer> seen = new HashSet<>();
    Set<Integer> dupes = new HashSet<>();
    for (int n : arr) {
        if (!seen.add(n)) dupes.add(n);
    }
    return dupes;
}

// Approach 2: Sort + scan — O(n log n), O(1) extra space
Arrays.sort(arr);
for (int i = 1; i < arr.length; i++) {
    if (arr[i] == arr[i-1]) /* duplicate found */;
}

Reverse a String without using built-in methods

Coding Round 1 
public static String reverse(String s) {
    char[] chars = s.toCharArray(); // toCharArray is usually allowed
    int left = 0, right = chars.length - 1;
    while (left < right) {
        char temp = chars[left];
        chars[left++] = chars[right];
        chars[right--] = temp;
    }
    return new String(chars);
}
// Pure manual: build new string char-by-char from end
// "hello" → "olleh"

Microservices Round 2

Monolith: single deployable unit, shared DB, simpler to start, harder to scale individual features. Microservices: independently deployable services, own databases, team autonomy, fault isolation — but adds network complexity, distributed tracing, and eventual consistency challenges.

What is API Gateway and why is it needed?

Microservices Round 2

Single entry point for all client requests (web, mobile). Routes to appropriate microservice. Provides:

  • Routing: /api/orders → Order Service
  • Authentication/Authorization: JWT validation once at gateway
  • Rate limiting: protect backend from abuse
  • SSL termination, load balancing, request logging
  • Response aggregation: combine multiple service calls into one response

Tools: Spring Cloud Gateway, Kong, AWS API Gateway. Clients never call microservices directly — reduces coupling and centralizes cross-cutting concerns.

How do you handle service failures in Microservices?

Resilience Round 2
  • Circuit Breaker: stop calling failing service, fail fast (Resilience4j)
  • Retry with backoff: transient failures (network blip)
  • Fallback: return cached/default response when downstream is down
  • Timeout: don't wait forever — set per-call timeouts
  • Bulkhead: isolate thread pools so one slow service doesn't exhaust all threads
  • Async + events: decouple with Kafka so failure in notification doesn't block order
  • Health checks + auto-restart: Kubernetes liveness/readiness probes

Explain Circuit Breaker Pattern

Resilience Round 2

Closed (normal) → Open (fail fast after threshold failures) → Half-Open (test one request). Prevents cascading failures when a downstream service is down. Implement with Resilience4j or Istio outlier detection.

Explain Database Indexing

SQL Round 2

An index is a separate data structure (typically B-Tree) that stores sorted column values + pointer to row — avoids full table scan.

  • Clustered index: determines physical row order (one per table — usually PK)
  • Non-clustered index: separate structure pointing to rows (can have many)
  • Composite index: multiple columns — follows leftmost prefix rule
CREATE INDEX idx_emp_dept_sal ON employees(department_id, salary);
-- Speeds: WHERE department_id = 5 ORDER BY salary
-- Won't help: WHERE salary > 50000 alone (salary is second column)

Trade-off: faster reads, slower writes (index must update on INSERT/UPDATE/DELETE), extra storage.

How would you optimize a slow-running query?

SQL Round 2

Run EXPLAIN ANALYZE → check for Seq Scan → add indexes → avoid SELECT * → rewrite correlated subqueries as JOINs → paginate with keyset instead of OFFSET → update table statistics → consider materialized views for heavy aggregations.

What is Redis and why do we use it?

Caching Round 2

Redis is an in-memory key-value store — extremely fast (sub-millisecond reads). Used for:

  • Caching: cache DB query results, API responses (TTL-based expiry)
  • Session storage: distributed session across multiple app instances
  • Rate limiting: sliding window counters
  • Pub/Sub: lightweight messaging
  • Leaderboards, real-time counters with sorted sets

In Spring Boot: spring-boot-starter-data-redis + @Cacheable or RedisTemplate.

Explain Cache-Aside Pattern

Caching Round 2
READ: App → check cache → hit? return → miss? read DB → write to cache → return WRITE: App → update DB → invalidate/delete cache entry (don't update cache on write — avoids stale data race)

Most common caching pattern. Application manages cache explicitly. Alternative: Read-Through (cache loads from DB itself), Write-Through (write to cache + DB together).

public Product getProduct(Long id) {
    Product cached = redis.get("product:" + id);
    if (cached != null) return cached;
    Product fromDb = repo.findById(id).orElseThrow();
    redis.set("product:" + id, fromDb, TTL_5_MIN);
    return fromDb;
}

What is Kafka?

Messaging Round 2

Distributed event streaming platform. Producers write messages to topics (partitioned logs). Consumers read at their own pace. Messages retained on disk (configurable retention) — replayable. High throughput, fault-tolerant, used for event-driven microservices, log aggregation, stream processing.

→ Kafka vs RabbitMQ in Microservices section

How does Kafka ensure message durability?

Kafka Round 2
  • Replication: each partition replicated across brokers (replication factor, e.g., 3)
  • Leader + ISR (In-Sync Replicas): leader handles writes; followers replicate; ack only after replicas confirm
  • Producer acks: acks=all — wait for all ISR replicas before confirming
  • Persistent storage: messages written to disk (sequential I/O), not just memory
  • Consumer offsets: committed offset tracking — replay from any point

Trade-off: higher durability settings (acks=all, min.insync.replicas=2) = lower throughput but no message loss on broker failure.

Design a URL Shortener System

System Design Round 2

Requirements: Given long URL → return short code. Given short code → redirect to long URL. High read volume, moderate writes.

Core components:

  • Encoding: Base62 (a-z, A-Z, 0-9) — 6 chars = 62⁶ ≈ 56 billion URLs. Or hash (MD5) truncated + collision check.
  • Storage: DB table (short_code PK, long_url, created_at, expiry) + Redis cache for hot redirects.
  • API: POST /shorten {url} → {shortUrl}; GET /{code} → 302 redirect.
  • Scale: read-heavy → cache 80% redirects in Redis; DB sharded by short_code hash.
  • Analytics: async click tracking via Kafka.
// Base62 encode auto-increment ID
char[] BASE62 = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ".toCharArray();
String encode(long id) {
    StringBuilder sb = new StringBuilder();
    while (id > 0) { sb.append(BASE62[(int)(id % 62)]); id /= 62; }
    return sb.reverse().toString();
}

What are Virtual Threads in Java 21?

Java 21 Round 2

Lightweight threads managed by the JVM, not the OS. Millions can run concurrently vs thousands of platform threads. Ideal for I/O-bound workloads (HTTP calls, DB queries) where threads mostly wait.

// Create virtual thread (Java 21+)
Thread.startVirtualThread(() -> callExternalApi());

// ExecutorService with virtual threads
ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor();
executor.submit(() -> processRequest());

Spring Boot 3.2+ supports virtual threads via spring.threads.virtual.enabled=true — Tomcat handles each request on a virtual thread. Not for CPU-bound work (use platform threads + limited pool).

How does ConcurrentHashMap achieve thread safety?

Concurrency Round 2

Java 8+ ConcurrentHashMap:

  • Reads: mostly lock-free via volatile node references
  • Writes: synchronized on individual bucket head (not whole map)
  • Null rejection: no null keys/values — avoids ambiguity in concurrent get/put
  • CAS operations: for counter updates and some insertions
  • Size computation: striped counter cells to avoid global lock

Result: much higher concurrency than Collections.synchronizedMap which locks the entire map on every operation.
