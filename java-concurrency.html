Core Java and Concurrency

Fundamental Java concepts covering collections, JVM memory, thread safety, and concurrent programming primitives.

What is the internal difference between HashMap and ConcurrentHashMap?

Collections Thread Safety

HashMap is a non-thread-safe hash table implementation. Internally it maintains an array of buckets. When you call put(key, value), the key's hash code is computed, mapped to a bucket index, and the entry is stored in a linked list (or red-black tree if the bucket exceeds 8 nodes in Java 8+). HashMap allows one null key and multiple null values. Its iterator is fail-fast — it throws ConcurrentModificationException if the map is structurally modified during iteration.

ConcurrentHashMap is designed for concurrent access without locking the entire map. In Java 8+, it uses a combination of:

  • CAS (Compare-And-Swap) for lock-free reads and some writes
  • Synchronized blocks on individual bucket heads for write operations
  • Volatile reads to ensure visibility across threads

It does not allow null keys or values (to avoid ambiguity in concurrent scenarios). Its iterator is weakly consistent — it reflects the state at some point during iteration and never throws ConcurrentModificationException.

Explain the JVM Memory Model: Heap, Stack, and Metaspace

JVM Memory
┌─────────────────────────────────────────┐ │ Metaspace (Java 8+) │ Class metadata, static vars ├─────────────────────────────────────────┤ │ Heap (shared across all threads) │ │ ├─ Young Generation │ │ │ ├─ Eden Space │ New objects allocated here │ │ └─ Survivor Spaces (S0, S1) │ Objects surviving minor GC │ └─ Old Generation (Tenured) │ Long-lived objects ├─────────────────────────────────────────┤ │ Stack (per thread) │ Method frames, local variables ├─────────────────────────────────────────┤ │ PC Register + Native Method Stack │ └─────────────────────────────────────────┘

Stack: Each thread gets its own stack. When a method is called, a new stack frame is pushed containing local variables, operand stack, and reference to the calling frame. Primitive types and object references (not the objects themselves) live here. Memory is automatically reclaimed when the method returns. Stack overflow occurs with deep recursion.

Heap: Shared memory region where all Java object instances and arrays are allocated. Managed by the Garbage Collector. Divided into Young Gen (Eden + Survivor spaces for short-lived objects) and Old Gen (long-lived objects promoted after surviving multiple GC cycles). OutOfMemoryError occurs when the heap is exhausted.

Metaspace: Replaced PermGen in Java 8. Stores class metadata — class definitions, method bytecode, constant pools, static variables. Grows dynamically based on application needs (unlike the fixed-size PermGen). Class unloading is possible when ClassLoaders are garbage collected.

What are Atomic Variables, and where are they commonly used?

Concurrency CAS

Atomic variables (AtomicInteger, AtomicLong, AtomicReference, AtomicBoolean, etc.) live in java.util.concurrent.atomic. They provide thread-safe operations on a single variable without using synchronized, leveraging hardware-level CAS (Compare-And-Swap) instructions.

CAS works by: read current value → compute new value → atomically swap only if current value hasn't changed → retry if it has. This is lock-free and avoids thread blocking.

AtomicInteger requestCount = new AtomicInteger(0);

// Thread-safe increment without synchronized
requestCount.incrementAndGet();

// Compare-and-set: set to 10 only if current value is 5
requestCount.compareAndSet(5, 10);

Common use cases:

  • Metrics counters and statistics in high-throughput systems
  • Status flags and state machines (e.g., shutdown signals)
  • Building lock-free data structures (ConcurrentLinkedQueue)
  • Sequence generators (unique ID generation without locks)
  • Implementing spin locks and non-blocking algorithms

Important: volatile does NOT make compound operations like count++ atomic. count++ is read-modify-write (three steps) and can race between threads even with volatile.

Use volatile when: A single variable is written by one thread and read by many (status flags, configuration toggles). Use synchronized when: You need to protect a critical section with multiple statements or compound operations.

How do you mitigate performance bottlenecks caused by synchronization?

Performance
  • Minimize lock scope: Lock only the critical section, not the entire method. Hold locks for the shortest time possible.
  • Finer-grained locking: Use ConcurrentHashMap instead of wrapping HashMap with synchronized. Lock per bucket/segment rather than the whole structure.
  • Lock-free alternatives: Replace locks with Atomic* variables and CAS operations for simple counters and flags.
  • ReadWriteLock: Allow multiple concurrent readers but exclusive writers — ideal for read-heavy workloads.
  • Immutable objects: If data doesn't change after creation, no synchronization is needed. Use final fields and defensive copies.
  • ThreadLocal: Give each thread its own copy of mutable state to eliminate sharing entirely.
  • Striped locking: Partition data into independent segments, each with its own lock (e.g., Guava's Striped).
  • Async/message passing: Use BlockingQueue or actor patterns to avoid shared mutable state.
  • Lock ordering: Always acquire locks in a consistent order to prevent deadlocks that cause threads to block indefinitely.

What is ThreadLocal, and how can it lead to memory leaks?

ThreadLocal Memory Leak

ThreadLocal provides thread-local variables — each thread accessing the variable gets its own independently initialized copy. Internally, each thread holds a ThreadLocalMap where keys are ThreadLocal instances and values are the stored objects.

private static final ThreadLocal<UserContext> context =
    ThreadLocal.withInitial(UserContext::new);

// Set at request start
context.set(new UserContext(userId, roles));

// Get anywhere in the same thread
UserContext ctx = context.get();

// MUST clean up
context.remove();
Memory leak mechanism: In thread pools (Tomcat, ExecutorService), threads are reused indefinitely. If you call set() with a large object (e.g., a UserContext with DB connections) and never call remove(), that object remains referenced by the thread's ThreadLocalMap forever — even after the request completes. Over time, this causes heap exhaustion.

Prevention: Always call threadLocal.remove() in a finally block, especially in servlet filters, interceptors, and thread-pool task wrappers.

What is a BlockingQueue, and where is it used in real-world applications?

Producer-Consumer

A BlockingQueue is a thread-safe queue that blocks the calling thread when:

  • put(element) is called on a full queue (bounded queues)
  • take() is called on an empty queue

This blocking behavior makes it ideal for coordinating producer-consumer workflows without busy-waiting or manual wait/notify.

Implementations:

  • ArrayBlockingQueue — bounded, array-backed, single lock
  • LinkedBlockingQueue — optionally bounded, linked nodes, two-lock design
  • PriorityBlockingQueue — unbounded, priority-ordered
  • DelayQueue — elements available only after a delay period
  • SynchronousQueue — zero capacity, direct handoff

Real-world uses: Thread pool work queues (ThreadPoolExecutor uses one internally), log aggregation pipelines, order processing systems, backpressure in streaming data pipelines, and decoupling API request handling from background processing.

What is a Functional Interface?

Java 8+ Lambda

A functional interface has exactly one abstract method (SAM — Single Abstract Method). Annotated with @FunctionalInterface (optional but recommended — compiler validates the rule).

@FunctionalInterface
interface Calculator {
    int operate(int a, int b);
}

Calculator add = (a, b) -> a + b;
Calculator multiply = (a, b) -> a * b;

// Built-in examples: Runnable, Callable, Comparator,
// Predicate, Function, Consumer, Supplier

Functional interfaces enable lambda expressions and method references. Java 8 added java.util.function package with common patterns for streams and optional composition.

Different ways to create a Thread and implement using Runnable

Thread 
// 1. Extend Thread class
class MyThread extends Thread {
    public void run() { System.out.println("Running"); }
}
new MyThread().start();

// 2. Implement Runnable (preferred over extending Thread)
class MyTask implements Runnable {
    public void run() { System.out.println("Task"); }
}
new Thread(new MyTask()).start();

// 3. Lambda (Runnable is a functional interface)
new Thread(() -> System.out.println("Lambda")).start();

// 4. Callable + Future (returns a result)
ExecutorService pool = Executors.newSingleThreadExecutor();
Future<String> future = pool.submit(() -> "result");

// 5. ExecutorService (production approach — see below)
Executors.newFixedThreadPool(4).submit(() -> doWork());

Always call start(), not run() directly — start() creates a new thread; run() executes in the current thread.

Thread

What is ExecutorService?

Thread Pool

ExecutorService is a higher-level API from java.util.concurrent that manages a pool of worker threads, a task queue, and thread lifecycle — so you submit work instead of creating raw threads.

ExecutorService executor = Executors.newFixedThreadPool(4);

// Runnable — no return value
executor.submit(() -> sendEmail(orderId));

// Callable — returns Future
Future<BigDecimal> future = executor.submit(() -> calculateTotal(orderId));
BigDecimal total = future.get(5, TimeUnit.SECONDS);

// Graceful shutdown
executor.shutdown();
executor.awaitTermination(30, TimeUnit.SECONDS);

Common pool types: newFixedThreadPool(n), newCachedThreadPool(), newSingleThreadExecutor(), newScheduledThreadPool(n). In Spring Boot, prefer @Async with a configured TaskExecutor bean over raw ExecutorService in business code.
