How does HashMap work internally?

Core Java 5+ YOE

Array of buckets. put(key, value): compute hash(key) ^ (hash >>> 16) → bucket index. Collision: linked list (Java 7) or tree if bucket > 8 nodes (Java 8+). Load factor 0.75 triggers resize (2× capacity, rehash).

// Contract: equal keys MUST have same hashCode
// Unequal hashCodes CAN collide (handled by equals() chain in bucket)
static final int hash(Object key) {
    int h = key.hashCode();
    return h ^ (h >>> 16); // spread high bits into index
}

Why interviewers ask: Leads to equals/hashCode, concurrency issues, and why HashMap is not thread-safe.

HashMap vs ConcurrentHashMap

Core Java

HashMap: not thread-safe, allows null key/value, fail-fast iterator. ConcurrentHashMap: bucket-level locking + CAS (Java 8+), no nulls, weakly consistent iterator, scales under concurrent reads/writes. Never use synchronized HashMap wrapper in high-throughput trading/cache paths.

equals() vs hashCode()

Core Java

equals(): logical equality. hashCode(): bucket index hint for hash-based collections.

  • If a.equals(b) then a.hashCode() == b.hashCode() — mandatory contract
  • Equal hashCodes do NOT imply equal objects (collisions OK)
  • Override both together — breaking contract breaks HashMap/HashSet silently
@Override public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof Trade t)) return false;
    return tradeId.equals(t.tradeId);
}
@Override public int hashCode() { return Objects.hash(tradeId); }

String vs StringBuilder vs StringBuffer

Core Java

String: immutable, thread-safe, string pool. StringBuilder: mutable, not synchronized, use in single-threaded loops. StringBuffer: mutable, synchronized on every method — legacy, slower. In banking report generation with heavy concatenation, StringBuilder in tight loops avoids O(n²) object creation.

Comparable vs Comparator

Core Java

Comparable<T>: natural ordering inside the class (compareTo) — one sort order only. Comparator<T>: external, multiple sort strategies without modifying the class.

// Comparable — built into Trade
class Trade implements Comparable<Trade> {
    public int compareTo(Trade o) { return this.tradeDate.compareTo(o.tradeDate); }
}
// Comparator — sort by notional without changing Trade
Comparator<Trade> byNotional = Comparator.comparing(Trade::getNotional).reversed();
trades.sort(byNotional);

Immutable Class — How to create one?

Core Java

Final class, final fields, no setters, defensive copy in constructor, unmodifiable getters. Critical in finance for thread-safe DTOs passed across threads without locks.

What is the String Constant Pool?

Core Java

Special area in heap (Java 7+ moved from PermGen) storing unique string literals. String s = "hello" reuses pool reference; new String("hello") creates heap object outside pool.

String a = "trade";      // pool
String b = "trade";      // same reference
String c = new String("trade"); // new object
System.out.println(a == b);     // true
System.out.println(a == c);     // false
System.out.println(a.equals(c)); // true

intern() adds string to pool — use sparingly; uncontrolled interning caused PermGen OOM in older JVMs.

Deep Copy vs Shallow Copy

Core Java

Shallow: copies field values — object references shared. Deep: recursively copies nested objects — fully independent clone. Matters when passing trade objects between threads or caching — shallow copy + mutable nested List = subtle bugs.

Multithreading

Locks, futures, coordination, thread pools

volatile keyword

Multithreading

Guarantees visibility (reads/writes go to main memory) and prevents instruction reordering for the variable. Does NOT make count++ atomic. Use for status flags: volatile boolean shutdownRequested. For compound operations use AtomicInteger or synchronized.

Future vs CompletableFuture

Multithreading

Future: blocking get(), no chaining. CompletableFuture: non-blocking pipelines — thenApply, thenCombine, allOf, exception handling with exceptionally. Use CompletableFuture to parallelize independent risk calculations then aggregate.

Deadlock and how to prevent it

Multithreading

Thread A holds lock1, waits for lock2; Thread B holds lock2, waits for lock1. Prevention:

  • Lock ordering — always acquire locks in same global order
  • tryLock with timeout — back off and retry
  • Reduce lock scope and duration
  • Use concurrent collections instead of nested synchronized blocks
  • Detect: jstack, thread dump — "Found one Java-level deadlock"

ExecutorService internals

Multithreading

ThreadPoolExecutor: core pool → queue (BlockingQueue) → max pool → RejectedExecutionHandler. Worker threads loop: pull task from queue, run, repeat. submit() wraps Runnable in FutureTask. Shutdown: shutdown() stops new tasks; awaitTermination() waits for completion.

new ThreadPoolExecutor(
    corePoolSize, maxPoolSize, keepAlive,
    TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(queueCapacity),
    threadFactory,
    new CallerRunsPolicy() // backpressure when saturated
);

Thread Pool tuning

Multithreading
  • CPU-bound: pool size ≈ cores + 1
  • I/O-bound: pool size ≈ cores × (1 + wait/compute ratio) — often 2×–4× cores
  • Bounded queue prevents unbounded memory growth under spike
  • Monitor: queue depth, active threads, rejection count via Micrometer
  • Separate pools for batch vs online API — bulkhead isolation

Producer-Consumer problem

Multithreading

Producers add to shared buffer; consumers remove. Solved with BlockingQueue — put() blocks when full, take() when empty. Used in trade event pipelines: market data feed (producer) → queue → risk engine (consumer).

BlockingQueue<TradeEvent> queue = new ArrayBlockingQueue<>(1000);
// Producer
queue.put(event);
// Consumer
TradeEvent e = queue.take();

Java Memory Model (JMM)

Multithreading

Specifies when writes by one thread are visible to another. Without synchronization/volatile, threads may see stale cached values. Happens-before rules: unlock→lock, volatile write→read, thread start/join. Critical for double-checked locking and lazy singleton initialization.

JVM

Memory, GC, class loading, performance

Heap vs Stack

JVM

Stack: per-thread, stores frames and local refs — fast, auto-freed. Heap: shared, all objects live here — GC managed. StackOverflowError vs OutOfMemoryError:heap space. Large collections and cached trade data → heap; method calls → stack.

Metaspace vs PermGen

JVM

PermGen (Java 7 and earlier): fixed-size, class metadata — PermGen OOM common with dynamic proxies (Spring/Hibernate). Metaspace (Java 8+): native memory, grows dynamically, class unloading when ClassLoader GC'd. Still can OOM if class loaders leak (hot redeploy without restart).

Garbage Collection algorithms

JVM
  • Serial: single thread, small apps
  • Parallel (Throughput): multi-thread Young/Old collection — batch jobs
  • G1: divides heap into regions, predictable pauses — default Java 9+
  • ZGC / Shenandoah: sub-ms pauses, large heaps (Java 17+)

Minor GC (Eden full) → Survivor → promoted to Old Gen → Major/Full GC when Old Gen full.

G1 GC vs Parallel GC

JVM

Memory Leak vs Memory Overflow

JVM

Leak: objects no longer needed but still referenced (static caches, ThreadLocal not cleared, unclosed connections) — heap grows until OOM. Overflow: legitimate data exceeds allocated heap (-Xmx). Fix leak: heap dump + MAT/VisualVM dominator tree. Fix overflow: increase -Xmx or reduce data retained.

Class Loading Process

JVM

Loading → Linking (verify, prepare, resolve) → Initialization (static blocks). Delegation model: Bootstrap → Platform → Application class loaders. Child asks parent first — prevents duplicate core classes. Custom ClassLoaders in app servers and Spring Boot nested jars (LaunchedURLClassLoader).

How JVM handles object creation?

JVM

new Trade(): class check → allocate memory on Eden (TLAB — thread-local allocation buffer for lock-free fast path) → zero memory → set header → init fields → invoke constructor. Most objects die young — collected in minor GC. Long-lived trade caches promote to Old Gen.

Spring Boot & Microservices

DI, auto-config, resilience, distributed systems

Bean Lifecycle

Spring Boot

Instantiate → populate → @PostConstruct → use → @PreDestroy. AOP proxy applied in BeanPostProcessor.afterInitialization. Know this to debug why @Transactional doesn't work (proxy not created) or init order issues.

Dependency Injection

Spring Boot

IoC container creates and wires beans. Constructor injection preferred — immutable, testable. Spring resolves by type; @Qualifier/@Primary for ambiguity. At banks, DI enables swapping mock pricing engines in tests vs prod implementations.

→ DI internals

@Component vs @Service vs @Repository

Spring Boot

All @Component stereotypes. @Repository adds persistence exception translation. @Service marks business layer semantically. Functionally identical for scanning — choose for clarity and tooling.

@Controller vs @RestController

Spring Boot

@RestController = @Controller + @ResponseBody on every method. Returns JSON via HttpMessageConverter. @Controller returns view names for server-rendered pages. Trading UIs may mix both — REST API + legacy JSP/Thymeleaf admin.

Spring Boot Auto Configuration

Spring Boot

Classpath scanning of AutoConfiguration.imports + @ConditionalOn* rules. Only matching configs apply. Override with custom @Bean or exclude via @SpringBootApplication(exclude=...). Explain with example: JPA starter → DataSource + EntityManagerFactory auto-wired.

Exception Handling in Microservices

Spring Boot 
@RestControllerAdvice
public class GlobalHandler {
    @ExceptionHandler(TradeNotFoundException.class)
    public ResponseEntity<ErrorBody> handle(TradeNotFoundException ex) {
        return ResponseEntity.status(404).body(new ErrorBody("TRADE_NOT_FOUND", ex.getMessage()));
    }
}

Standardize error schema across services (code, message, traceId). Propagate correlation ID for distributed debugging. Map domain exceptions to HTTP status — never expose stack traces in prod.

Service Discovery

Microservices

Services register with registry (Eureka, Consul); clients resolve dynamic instances instead of hardcoded URLs. Enables scaling and rolling deploys. In Kubernetes, often replaced by native DNS + Services, but pattern remains same.

API Gateway

Microservices

Single entry: routing, auth (JWT/OAuth), rate limiting, SSL termination, request aggregation. Spring Cloud Gateway, Kong. Banks use gateway for WAF integration and centralized audit logging of all API access.

How do you investigate high CPU usage?

Production
  1. Confirm: top/htop, which JVM process
  2. jstack <pid> — take 3 dumps 30s apart, find threads consistently in RUNNABLE on same stack
  3. async-profiler or jcmd <pid> Thread.print for hot methods
  4. Check recent deploy, traffic spike, GC thrashing (high usr vs sys CPU)
  5. Common causes: infinite loop, regex catastrophic backtracking, excessive GC, crypto without hardware accel

How do you troubleshoot OutOfMemoryError?

Production
  • Identify type: Java heap space, Metaspace, GC overhead limit exceeded, Direct buffer memory
  • Enable heap dump on OOM: -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/logs/
  • Analyze with Eclipse MAT — dominator tree, leak suspects, largest retained sets
  • Check heap trend in Grafana before crash — gradual climb = leak; sudden spike = traffic/data burst
  • Fix: remove leak source or tune -Xmx/GC; never just increase heap without root cause

How do you analyze slow APIs?

Production
  1. APM trace (Dynatrace, AppDynamics, Zipkin) — identify slow span: DB, external call, or app code
  2. DB: slow query log, EXPLAIN ANALYZE, missing index, N+1 from Hibernate
  3. External: timeout config, circuit breaker state, downstream latency
  4. App: thread pool saturation, lock contention, large payload serialization
  5. Compare p50 vs p99 — tail latency often reveals contention outliers

What monitoring tools have you used?

Production

Mention what you know: Prometheus + Grafana (metrics), ELK/Splunk (logs), Jaeger/Zipkin (tracing), Micrometer + Spring Actuator, Dynatrace/AppDynamics (APM), PagerDuty/Opsgenie (alerting). Banks often require all three pillars: metrics, logs, traces with correlation ID tying them together.

How do you perform Root Cause Analysis (RCA)?

Production
  1. Timeline: when detected, when started, deploy correlation
  2. Impact: users affected, financial/SLA impact
  3. 5 Whys: dig past symptoms to systemic cause
  4. Evidence: logs, metrics, thread dumps, change records
  5. Action items: fix, test, monitor, runbook update — preventive not just reactive
  6. Blameless postmortem — focus on process and system improvements
The difference between selected and rejected: Often not coding ability — it's how deeply you understand the why behind HashMap buckets, GC pause targets, saga vs 2PC, and how you'd debug a 3 AM OOM in production.
