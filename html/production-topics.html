Java Streams & Collections

flatMap, ConcurrentHashMap, Set internals, Functional Interface

Java Streams — flatMap explained

Streams

map transforms each element 1:1. flatMap transforms each element into a stream of results and flattens all streams into one.

// map: List<Order> → List<List<LineItem>> (nested)
orders.stream().map(Order::getItems).toList();

// flatMap: List<Order> → List<LineItem> (flattened)
List<LineItem> allItems = orders.stream()
    .flatMap(order -> order.getItems().stream())
    .toList();

// Real use: split sentences into words
List<String> words = sentences.stream()
    .flatMap(s -> Arrays.stream(s.split(" ")))
    .toList();

// Optional chaining — avoid nested if-present checks
user.flatMap(User::getAddress)
    .flatMap(Address::getZipCode)
    .orElse("00000");

Interview tip: Use flatMap when the mapping function returns a collection/stream per element, or when chaining Optional values.

ConcurrentHashMap — production use cases

Collections

Thread-safe map without locking the entire structure. Use for shared caches, rate-limit counters, session stores in multi-threaded servers.

// Thread-safe cache with computeIfAbsent (atomic get-or-load)
private final ConcurrentHashMap<Long, Product> cache = new ConcurrentHashMap<>();

public Product getProduct(Long id) {
    return cache.computeIfAbsent(id, key -> loadFromDb(key));
}

// Atomic counter per API key (rate limiting)
ConcurrentHashMap<String, AtomicInteger> requestCounts = new ConcurrentHashMap<>();
requestCounts.computeIfAbsent(clientId, k -> new AtomicInteger(0))
             .incrementAndGet();

Multithreading & Concurrency

wait/notify, race conditions, atomicity, locking

wait() and notify() — how they work

Concurrency

wait(), notify(), and notifyAll() are called on an object's monitor lock — must be inside a synchronized block on that same object.

class BoundedBuffer {
    private final Queue<Task> queue = new LinkedList<>();
    private final int capacity = 10;

    synchronized void produce(Task task) throws InterruptedException {
        while (queue.size() == capacity) {
            wait(); // release lock, wait until consumer notifies
        }
        queue.add(task);
        notifyAll(); // wake waiting consumers
    }

    synchronized Task consume() throws InterruptedException {
        while (queue.isEmpty()) {
            wait();
        }
        Task task = queue.poll();
        notifyAll(); // wake waiting producers
        return task;
    }
}

Production note: Prefer BlockingQueue (ArrayBlockingQueue) over manual wait/notify — less error-prone, handles spurious wakeups with built-in conditions.

Race condition handling

Concurrency

A race condition occurs when multiple threads read/write shared mutable state and the outcome depends on timing.

// RACE: two threads both read count=5, both write 6
private int count = 0;
public void increment() { count++; } // NOT atomic (read-modify-write)

// Fix 1: synchronized
public synchronized void increment() { count++; }

// Fix 2: AtomicInteger
private AtomicInteger count = new AtomicInteger(0);
public void increment() { count.incrementAndGet(); }

// Fix 3: eliminate shared state — immutable objects, ThreadLocal

Detection: intermittent bugs under load, works in dev but fails in prod. Use stress tests, thread dump analysis, and tools like FindBugs/SpotBugs.

Spring Boot & Microservices

@Async, Retry, Circuit Breaker, Stateless vs Stateful

@Async — async processing in Spring Boot

Spring Boot 
@EnableAsync
@SpringBootApplication
public class App { }

@Service
public class NotificationService {
    @Async
    public CompletableFuture<Void> sendEmail(Order order) {
        emailClient.send(order.getCustomerEmail(), "Order confirmed");
        return CompletableFuture.completedFuture(null);
    }
}

Internals: Spring creates a JDK/CGLIB proxy. Calling sendEmail() from another bean submits work to a TaskExecutor thread pool and returns immediately. Self-invocation (this.sendEmail()) bypasses proxy — no async!

Configure pool: @Bean(name = "taskExecutor") with core/max pool size. Propagate MDC/security context with TaskDecorator.

Retry pattern in Spring Boot

Resilience 
// Resilience4j or Spring Retry
@Retry(name = "paymentService", fallbackMethod = "paymentFallback")
public PaymentResult charge(PaymentRequest req) {
    return paymentClient.charge(req); // may throw on transient failure
}

public PaymentResult paymentFallback(PaymentRequest req, Exception ex) {
    return PaymentResult.pending("Queued for retry");
}

// application.yml
resilience4j.retry:
  instances:
    paymentService:
      maxAttempts: 3
      waitDuration: 500ms
      exponentialBackoffMultiplier: 2

Retry only idempotent operations or those with idempotency keys. Never blind-retry non-idempotent payment without dedup logic.

Circuit Breaker — production implementation

Resilience 
@CircuitBreaker(name = "inventoryService", fallbackMethod = "inventoryFallback")
public StockInfo checkStock(Long productId) {
    return inventoryClient.getStock(productId);
}

public StockInfo inventoryFallback(Long productId, CallNotPermittedException ex) {
    return StockInfo.cached(productId); // serve stale cache when circuit open
}

States: CLOSED → OPEN (after failure threshold) → HALF-OPEN (probe). Prevents thread exhaustion calling a dead service. Combine with timeout + bulkhead for full resilience stack.

Hibernate & Database

N+1 problem, Lazy vs Eager, Connection Pooling, Query Optimization

The N+1 Query Problem

Hibernate

One query loads N parent entities, then N additional queries load each child — total N+1 queries instead of 1-2.

// Triggers N+1: 1 query for orders + N queries for each order's items
List<Order> orders = orderRepo.findAll(); // SELECT * FROM orders
orders.forEach(o -> o.getItems().size()); // SELECT * FROM items WHERE order_id=? (×N)

// Fix 1: JOIN FETCH
@Query("SELECT o FROM Order o JOIN FETCH o.items")
List<Order> findAllWithItems();

// Fix 2: @EntityGraph
@EntityGraph(attributePaths = {"items"})
List<Order> findAll();

// Fix 3: DTO projection — fetch only needed columns
@Query("SELECT new OrderDto(o.id, o.total) FROM Order o")
List<OrderDto> findOrderSummaries();

Enable spring.jpa.show-sql=true or Hibernate statistics in dev to catch this. Use datasource proxy (p6spy) to count queries per request.

Lazy vs Eager Loading

Hibernate
Lazy (default for @OneToMany, @ManyToMany)Eager (default for @ManyToOne, @OneToOne)
Child loaded on first access via proxyChild loaded immediately with parent
Better performance — load only what's neededRisk of loading entire object graph unintentionally
Requires open session or JOIN FETCH in queryCan cause cartesian product with multiple collections
LazyInitializationException: Accessing lazy collection outside a transaction (after service method returns). Fix: @Transactional on service, use DTO, or JOIN FETCH in query. Avoid switching everything to EAGER — it causes worse performance at scale.

Connection Pooling — HikariCP in Spring Boot

Database

Creating a DB connection is expensive (TCP handshake, auth, memory). Connection pool maintains ready-to-use connections.

spring.datasource.hikari.maximum-pool-size=20
spring.datasource.hikari.minimum-idle=5
spring.datasource.hikari.connection-timeout=30000
spring.datasource.hikari.idle-timeout=600000
spring.datasource.hikari.max-lifetime=1800000
  • Pool too small: threads block waiting for connection → timeouts under load
  • Pool too large: overwhelms DB max connections, GC pressure
  • Rule of thumb: pool size ≈ number of CPU cores × 2 for OLTP (adjust with profiling)
  • Monitor: active connections, pending threads, connection acquire time via Actuator/Micrometer

Hibernate & SQL Query Optimization

Performance
  • Use pagination — Pageable with LIMIT, never load full table
  • Batch inserts — spring.jpa.properties.hibernate.jdbc.batch_size=50
  • Avoid SELECT * — use projections/DTOs for list views
  • Second-level cache (Redis/Ehcache) for read-heavy reference data
  • Index foreign keys and WHERE/ORDER BY columns
  • Use native query or JPQL with explicit joins instead of lazy loading in loops
  • @QueryHints(@QueryHint(name = "org.hibernate.readOnly", value = "true")) for read-only queries

System Design Scenarios

High-concurrency inventory and overselling prevention

Only 1 stock left — 1000 users buying simultaneously. How do you prevent overselling?

System Design Concurrency

This is a classic inventory concurrency problem. Layer defenses from DB to application:

Layer 1 — Pessimistic locking (DB level):

-- PostgreSQL: lock the row until transaction commits
SELECT stock FROM products WHERE id = ? FOR UPDATE;
-- If stock > 0: UPDATE products SET stock = stock - 1 WHERE id = ?;
-- Commit. Other 999 threads block on FOR UPDATE, then see stock=0.

Layer 2 — Optimistic locking (version column):

@Entity
class Product {
    @Version
    private Long version; // Hibernate checks version on UPDATE
    private int stock;
}
// UPDATE fails if version changed → retry or return "sold out"

Layer 3 — Atomic Redis decrement:

// Lua script or DECR — atomic, sub-millisecond
Long remaining = redis.decr("stock:product:" + productId);
if (remaining < 0) { redis.incr(key); return SOLD_OUT; }
// proceed to create order async

Layer 4 — Queue-based ordering: Flash sale requests go to Kafka queue; single consumer processes sequentially — guarantees order, handles burst traffic.

Layer 5 — Idempotency: Each purchase request has idempotency key — duplicate clicks from same user don't create multiple orders.

Interview answer structure: Acknowledge race condition → explain layered approach → pick strategy based on scale (Redis for flash sales, DB lock for normal e-commerce) → mention monitoring and compensating transactions on payment failure.

Scaling factors — vertical vs horizontal

Performance
  • Vertical scaling: bigger CPU/RAM — simple but has ceiling and single point of failure
  • Horizontal scaling: more instances behind load balancer — requires stateless services
  • Database: read replicas, connection pooling, sharding for write scale
  • Async processing: offload heavy work to Kafka consumers — decouple response time from processing time
  • Rate limiting: protect backend from traffic spikes
  • Auto-scaling triggers: CPU > 70%, request latency p99, queue depth

Deployment versioning strategies

DevOps
  • Semantic versioning: MAJOR.MINOR.PATCH (1.4.2) — communicate breaking changes
  • Git SHA tagging: myapp:abc123f — trace exact code in production
  • Blue-Green: two identical environments; switch traffic instantly; fast rollback
  • Canary: route 5% traffic to new version; monitor errors/latency; gradually increase
  • Rolling update: Kubernetes replaces pods one-by-one — zero downtime with readiness probes

Common cloud migration challenges

Cloud
  • Latency: services in different AZs/regions — design for network partitions
  • Secrets management: never in code — use Vault, AWS Secrets Manager, K8s Secrets
  • Observability gap: distributed tracing (Jaeger/Zipkin) essential across microservices
  • Cost control: right-size instances, autoscale down, spot instances for batch jobs
  • Data residency/compliance: GDPR, data sovereignty requirements
  • Cold start: JVM apps slower to start than Node — use layered Docker jars, GraalVM native images
  • Vendor lock-in: abstract cloud-specific APIs behind interfaces where possible

Java Records — what they are and when to use them

Java 17

A record is an immutable data carrier — compiler auto-generates constructor, getters, equals(), hashCode(), toString().

// Before: 30 lines of boilerplate POJO
public record OrderDto(Long id, String status, BigDecimal total) { }

// Usage — same as a class
OrderDto dto = new OrderDto(1L, "SHIPPED", new BigDecimal("99.99"));
dto.id();    // accessor (not getId())
dto.total();

// Compact constructor — validation
public record Email(String value) {
    public Email {
        if (!value.contains("@")) throw new IllegalArgumentException("Invalid email");
    }
}

// Real-world use cases:
// 1. API response/request DTOs
// 2. Kafka event payloads
// 3. JDBC row mapping results
// 4. Map keys (immutable + correct equals/hashCode)
// 5. Multiple return values from methods
public record StockResult(boolean available, int remaining) { }

Limitations: cannot extend other classes (implicitly extend Record), fields are final, not ideal for JPA entities (use regular classes for mutable entity state). Perfect for immutable value objects and DTOs.

Other Java 17 features to mention: Sealed classes (restricted inheritance hierarchies), pattern matching for switch (preview), text blocks (""" multi-line strings).
