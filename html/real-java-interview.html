Real Java Interview Questions — Production Thinking

Java interviews don't reject you for syntax mistakes — they reject you for weak thinking around how Java behaves in real systems. These 20 questions test real understanding, not memorization.

REAL JAVA · NOT TEXTBOOK

How Java behaves under load, in production, and over time

JVM, threads, GC, collections, caching — the questions senior interviewers actually ask today.

JVM & Runtime Behavior
REAL 01

Why does a Java application slow down over time without throwing errors?

In simple words

Nothing crashes — but memory fills up, GC runs more often, connection pools get exhausted, caches grow unbounded, or thread pools queue up. Each request takes longer even though code didn't change.

What to tell the interviewer
  • Memory leak / heap pressure: objects retained in caches, maps, ThreadLocals — GC pauses grow.
  • Connection pool exhaustion: leaked JDBC connections — threads block waiting.
  • Thread pool queue growth: tasks pile up under sustained load.
  • Disk full: logs can't write — I/O blocks.
  • Metaspace creep: dynamic class generation (CGLIB, Groovy) fills metaspace.
  • Debug: GC logs, heap trend (Micrometer), thread dump count, connection pool metrics.
REAL 02

What causes OutOfMemoryError even when heap size looks sufficient?

In simple words

OOM isn't always "heap too small." Java has other memory areas — metaspace, direct memory, native memory, thread stacks — that can blow up while heap looks fine on a chart.

What to tell the interviewer
  • Metaspace OOM: too many classes loaded (hot redeploy, classloader leak).
  • Direct buffer OOM: Netty/NIO off-heap (-XX:MaxDirectMemorySize).
  • Unable to create native thread: thousands of threads × ~1 MB stack each.
  • Heap sized for average, not peak: traffic spike exceeds -Xmx.
  • Read the OOM message — it tells you which area failed.
  • Enable -XX:+HeapDumpOnOutOfMemoryError + analyze with MAT/Eclipse Memory Analyzer.
REAL 03

How does GC tuning impact application latency in production?

In simple words

GC stops application threads during collection (stop-the-world). Wrong heap size or wrong collector = frequent or long pauses → API p99 latency spikes even when average looks fine.

What to tell the interviewer
  • G1GC (Java 9+ default): targets pause time via -XX:MaxGCPauseMillis=200.
  • Young GC frequent = short pauses; Full GC = long pause — avoid Full GC in prod.
  • ZGC / Shenandoah: sub-ms pauses for latency-sensitive services (Java 17+).
  • Too small heap → GC thrashing (constant collection, high CPU).
  • Monitor p99 latency alongside GC pause metrics — correlate spikes.
  • Don't tune blindly — profile allocation rate first; often fix is code (object churn), not flags.
-Xms4g -Xmx4g -XX:+UseG1GC -XX:MaxGCPauseMillis=200
-Xlog:gc*:file=gc.log:time,uptime,level,tags
REAL 14

What happens if GC runs too frequently?

In simple words

App spends more time collecting garbage than doing work. CPU spikes, throughput drops, latency jitter increases — but no error is thrown.

What to tell the interviewer
  • GC thrashing: heap too small for allocation rate — constant Young GC.
  • Symptoms: high CPU, low business throughput, GC logs show back-to-back collections.
  • Fix: increase heap, reduce object allocation (reuse buffers, avoid boxing in loops), fix leaks.
  • Premature promotion: short-lived objects reach Old gen → expensive Full GC cycles.
REAL 16

How does JVM warm-up affect benchmark results?

In simple words

Cold JVM is slow — JIT hasn't compiled hot code yet. First few thousand requests are misleading. Production apps get faster after warm-up; benchmarks without warm-up lie.

What to tell the interviewer
  • C1 (fast compile) → C2 (aggressive optimize) tiers over time.
  • First run: interpreted bytecode; after 10K iterations: inlined hot methods.
  • Use JMH with warmup iterations — never benchmark cold start for steady-state perf.
  • K8s cold pod after scale-up: use readiness probe + synthetic warm-up traffic.
  • Class loading at startup also adds latency — Spring context init time matters.
REAL 09

How does JVM handle memory for threads vs objects?

In simple words

Objects live on the heap (shared by all threads). Each thread gets its own stack for local variables and method calls — not visible to other threads. Thread metadata also consumes native memory.

What to tell the interviewer
  • Heap: all new objects, shared — GC manages this.
  • Stack: per-thread, stores primitives and object references — no GC, freed when frame pops.
  • Default stack ~1 MB/thread (-Xss) — 10,000 threads ≈ 10 GB native memory.
  • Metaspace: class metadata (separate from heap since Java 8).
  • Race conditions happen when multiple threads mutate shared heap objects without sync.
REAL 20

What JVM flags are most useful during performance issues?

In simple words

When prod is slow or OOM, these flags give you evidence — not guesses. GC logs, heap dumps, native memory tracking, and flight recorder.

What to tell the interviewer
  • -Xlog:gc* — GC logging (Java 9+ unified logging).
  • -XX:+HeapDumpOnOutOfMemoryError — snapshot on OOM for MAT analysis.
  • -XX:NativeMemoryTracking=summary — thread/direct/metaspace breakdown.
  • jcmd <pid> VM.native_memory summary — live NMT report.
  • async-profiler / JFR — CPU flame graphs without heavy overhead.
  • jstack <pid> — thread dump for deadlocks and blocked threads.
java -Xms2g -Xmx2g -XX:+UseG1GC \
  -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp \
  -Xlog:gc*:file=gc.log:time,level,tags \
  -jar app.jar
Concurrency & Threads
REAL 04

Why is synchronized sometimes worse than Lock?

In simple words

synchronized is simpler but less flexible. ReentrantLock gives try-lock with timeout, interruptible waits, and multiple Condition queues — useful when you can't afford to block forever.

What to tell the interviewer
  • synchronized: no timeout — thread blocks until lock free (deadlock risk).
  • lock.tryLock(5, SECONDS) — fail fast instead of hanging prod request.
  • lock.lockInterruptibly() — respond to shutdown signals.
  • Condition — multiple wait sets (not just one intrinsic wait set).
  • Prefer synchronized for simple cases — less error-prone (auto-release).
  • ReentrantLock must unlock in finally — easy to leak lock if misused.
REAL 05

How do thread leaks happen and how do you detect them?

In simple words

Threads are created but never stopped — each one holds ~1 MB stack. Over days, thread count climbs until "unable to create native thread" or system slows down.

What to tell the interviewer
  • Causes: ExecutorService never shut down; new Thread per request; HTTP client threads; scheduled tasks without lifecycle management.
  • Detect: monitor thread count metric over time; jstack — count threads with same stack trace.
  • Actuator /actuator/metrics/jvm.threads.live — alert on steady growth.
  • Fix: use managed thread pools, @PreDestroy shutdown(), daemon threads for background only.
REAL 07

What happens if a thread dies while holding a lock?

In simple words

With synchronized, JVM releases the monitor when the thread terminates — waiting threads can proceed. With ReentrantLock left unlocked in a crash path, other threads block forever unless you used tryLock/timeout.

What to tell the interviewer
  • synchronized: lock released on thread death — JVM cleans up monitor.
  • ReentrantLock: NOT auto-released — must use finally { lock.unlock(); }.
  • Uncaught exception in synchronized block still releases lock (block exit).
  • Always prefer try/finally with explicit locks in production code.
REAL 13

How do you debug deadlocks in a live Java application?

In simple words

Two threads each hold a lock the other needs — both wait forever. Take a thread dump — JVM prints "Found one Java-level deadlock" with the exact threads and locks.

What to tell the interviewer
  • jstack <pid> or kill -3 <pid> — thread dump without stopping app.
  • Spring Actuator /actuator/threaddump — same output via HTTP.
  • Look for BLOCKED threads waiting on same monitor chain.
  • Fix: consistent lock ordering, timeout with tryLock, reduce lock scope.
REAL 15

Why do microservices fail due to thread pool misconfiguration?

In simple words

Too few threads → requests queue and timeout. Too many threads → overwhelm DB (more connections than pool allows), memory exhaustion, context switching overhead.

What to tell the interviewer
  • Tomcat default 200 threads × 5 services calling same DB with 20 connections = disaster.
  • Rule: max threads ≤ DB pool size for DB-bound services.
  • Blocking HTTP calls on limited pool → cascade timeout across microservices.
  • Fix: right-size pools, async/reactive for I/O, bulkhead pattern (Resilience4j).
Collections, Streams & APIs
REAL 10

What are real-world pitfalls of using parallel streams?

In simple words

Parallel streams use the common ForkJoinPool. Blocking I/O or DB calls inside parallel stream starves the entire JVM's parallel tasks. Small lists are slower due to fork overhead.

What to tell the interviewer
  • Default pool = ForkJoinPool.commonPool() — shared globally.
  • Never do blocking I/O inside .parallelStream().
  • Shared mutable state in forEach → race conditions.
  • Small collections (< 10K): parallel overhead > benefit.
// BAD — blocks common pool
orders.parallelStream().forEach(o -> restClient.call(o));
// GOOD — CPU-bound transform on large in-memory list
largeList.parallelStream().map(this::computeHash).toList();
REAL 11

How can improper equals/hashCode crash a production system?

In simple words

HashMap/HashSet rely on hashCode for bucket placement and equals for lookup. Break the contract → duplicates in sets, "lost" objects in maps, O(n) bucket chains — silent data corruption or sudden slowdown.

What to tell the interviewer
  • Mutable object as HashMap key — hash changes after insert → object unreachable.
  • Lombok @Data on entity with lazy field — wrong equals/hashCode.
  • Only override equals without hashCode — HashSet contains() breaks.
  • Fix: immutable keys, business ID in equals/hashCode only.
REAL 17

What are common mistakes with CompletableFuture in production?

In simple words

Async code that blocks, swallows exceptions, or hogs the common pool causes mysterious prod failures — timeouts, thread starvation, silent errors.

What to tell the interviewer
  • .get() without timeout — blocks calling thread forever.
  • Exceptions swallowed unless .exceptionally() or .handle() added.
  • Default supplyAsync() uses common pool — blocking I/O starves JVM.
  • Fix: dedicated executor; orTimeout(5, SECONDS) (Java 9+).
CompletableFuture.supplyAsync(() -> callPayment(), paymentExecutor)
    .orTimeout(5, TimeUnit.SECONDS)
    .exceptionally(ex -> { log.error("Payment failed", ex); return fallback; });
Production Behavior & Debugging
REAL 06

Why does high CPU usage occur even when traffic is low?

In simple words

CPU isn't only driven by user requests. Background jobs, GC loops, infinite retry logic, tight spin-waits, or a scheduled task gone wrong can peg CPU at 100% with zero traffic.

What to tell the interviewer
  • GC thrashing, runaway @Scheduled task, busy-wait loops, regex backtracking.
  • Debug: async-profiler flame graph, JFR, per-thread CPU in top -H.
REAL 08

Why do applications behave differently under load compared to local runs?

In simple words

Locally: one user, H2 database, no network. In prod: concurrent requests expose race conditions, pool limits, GC pressure, and latency that never appear on your machine.

What to tell the interviewer
  • Race conditions at 100+ parallel requests; pool limits; H2 vs PostgreSQL; cold cache after deploy.
  • Load test with Gatling/k6 before release.
REAL 12

Why does caching sometimes reduce performance instead of improving it?

In simple words

Cache adds a network hop, serialization cost, and complexity. Low hit rate or huge values can make you slower than hitting the DB once.

What to tell the interviewer
  • Low hit rate = Redis RTT + DB on every miss; thundering herd on expiry; large value serialization cost.
  • Measure hit rate and p99 with vs without cache.
REAL 18

Why does excessive logging slow down Java apps?

In simple words

Logging in hot paths does string building and sync I/O — even when DEBUG is off, if you use string concat instead of parameterized logs.

What to tell the interviewer
  • Use log.debug("User {} order {}", user, order) not string concat.
  • Async appenders (Logback AsyncAppender); don't log full JSON payloads in hot path.
REAL 19

How do memory leaks occur without static references?

In simple words

Objects stay alive if anything still references them — ThreadLocals, listeners, unbounded caches, classloader leaks — not just static fields.

What to tell the interviewer
  • ThreadLocal not removed in pooled threads; event listeners never unregistered; unbounded Guava/CHM cache.
  • Heap dump → MAT "Leak Suspects" → dominator tree.
How to stand out: Don't recite definitions — tell a story. "We had p99 latency spike every 30 seconds — GC logs showed promotion failure. We fixed a ThreadLocal leak in our request filter." That's what gets offers.
