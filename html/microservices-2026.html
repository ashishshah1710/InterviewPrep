2026 TREND · 3–5 YOE

Latest Java Microservices Interview Questions

32 questions across 6 categories — each with simple explanation and interview talking points.

➤ Core Java
CORE 01

Difference between Fail-Fast and Fail-Safe?

In simple words

Fail-fast throws error immediately if collection changes during iteration (ArrayList). Fail-safe works on a snapshot — no error but may miss fresh changes (ConcurrentHashMap iterator).

What to tell the interviewer
  • Fail-fast: ConcurrentModificationException — HashMap, ArrayList.
  • Fail-safe: weakly consistent — ConcurrentHashMap, CopyOnWriteArrayList.
  • Never modify collection while iterating except via iterator.remove().
CORE 02

How does ConcurrentHashMap work internally?

In simple words

Array of buckets. Reads mostly lock-free. Writes lock only one bucket. Java 8+ uses synchronized on bucket head + CAS — not the whole map.

What to tell the interviewer
  • No null key/value — avoids ambiguity concurrent get/put.
  • Treeify bucket when chain > 8 nodes.
  • Project: shared cache or rate-limit counters in API gateway.
CORE 03

What is a Daemon Thread?

In simple words

Background thread that dies when all user (non-daemon) threads finish — JVM exits without waiting for daemons.

What to tell the interviewer
  • thread.setDaemon(true) before start().
  • Examples: GC threads, Tomcat housekeeping — not for business logic you must complete.
  • Main thread is always non-daemon by default.
CORE 04

Comparable vs Comparator?

In simple words

Comparable = natural order inside class (compareTo). Comparator = external sort rules — multiple sorts without changing class.

What to tell the interviewer
  • Comparable: one sort order. Comparator: salary desc, then name asc.
  • Streams: .sorted(Comparator.comparing(...)).
employees.sort(Comparator.comparing(Employee::getSalary).reversed());
CORE 05

SOLID principles with real-time example?

In simple words

Five design rules for maintainable code — single responsibility, open/closed, Liskov, interface segregation, dependency inversion.

What to tell the interviewer
  • S: OrderService only orders — PaymentService handles payment.
  • O: Add UPI payment without editing OrderService — new PaymentStrategy impl.
  • L: Subclass honors parent contract — no surprise throws.
  • I: Split fat interface into Reader/Writer not one DatabaseGod.
  • D: Depend on PaymentGateway interface, inject Stripe/Mock impl.
CORE 06

Optional: isPresent() vs ifPresent()?

In simple words

isPresent() returns boolean — you manually unwrap. ifPresent(consumer) runs action only if value exists — no explicit get().

What to tell the interviewer
  • Prefer ifPresent / map / orElseThrow — avoid get() without check.
  • ifPresent avoids nested if (optional.isPresent()) { get() }.
optional.ifPresent(user -> sendEmail(user.getEmail()));
User u = optional.orElseThrow(() -> new NotFoundException());
CORE 07

How does JVM handle memory management?

In simple words

Objects on heap (Young + Old gen). GC removes unreferenced objects. Stack per thread for method calls. Metaspace for class metadata.

What to tell the interviewer
  • Minor GC on Eden; long-lived → Old gen; Major/Full GC when Old full.
  • Tune -Xmx, G1GC for microservices; watch GC logs in prod.
  • OOM types: heap, metaspace, direct memory.
➤ Spring Boot & Microservices
SPRING 01

How do you implement JWT authentication?

In simple words

User logs in → server signs JWT (header.payload.signature) with secret → client sends Bearer token on each request → filter validates signature and expiry.

What to tell the interviewer
  • Login endpoint returns JWT; store secret server-side only.
  • OncePerRequestFilter validates token before controller.
  • Spring Security 6 + OAuth2 Resource Server or custom JwtFilter.
  • Short expiry + refresh token for security.
@PostMapping("/login")
public TokenResponse login(@RequestBody LoginRequest req) {
    User u = authService.authenticate(req);
    return new TokenResponse(jwtService.generateToken(u));
}
// Filter: validate Authorization: Bearer <token>
SPRING 02

OAuth2 vs JWT?

In simple words

OAuth2 is an authorization framework (roles, scopes, token exchange). JWT is a token format — often used as the access token inside OAuth2.

What to tell the interviewer
  • OAuth2: Authorization Code flow with Keycloak/Auth0 — third-party login.
  • JWT: self-contained claims — stateless API auth between microservices.
  • Not either/or — OAuth2 often issues JWT access tokens.
SPRING 03

RestTemplate vs WebClient?

In simple words

RestTemplate = blocking sync HTTP — thread waits. WebClient = reactive/non-blocking — better for high concurrency I/O.

What to tell the interviewer
  • RestTemplate deprecated in Spring 5 — use WebClient for new code.
  • WebClient + virtual threads (Java 21) also works for sync-style with better scale.
  • Microservice calls: set timeouts, circuit breaker (Resilience4j).
WebClient client = WebClient.create("http://payment-service");
Mono<PaymentResult> result = client.get()
    .uri("/payments/{id}", id)
    .retrieve().bodyToMono(PaymentResult.class);
SPRING 04

How do microservices communicate?

In simple words

Sync: REST/gRPC when you need immediate answer. Async: Kafka/RabbitMQ when you can fire-and-forget or event-drive.

What to tell the interviewer
  • Sync: order checks inventory via REST — coupling + latency.
  • Async: OrderCreated event → payment/inventory consume independently.
  • Service discovery + API Gateway for sync routing.

→ Full event-driven workflow

SPRING 05

What is API Gateway and why is it required?

In simple words

Single entry for all clients — auth, rate limit, routing, SSL — microservices stay hidden behind it.

What to tell the interviewer
  • Spring Cloud Gateway, Kong, AWS API Gateway.
  • JWT validation once at edge — not in every service.
  • Route /orders → order-service, /payments → payment-service.
SPRING 06

How to implement Global Exception Handling?

In simple words

@RestControllerAdvice + @ExceptionHandler — one place maps all exceptions to consistent JSON error responses.

What to tell the interviewer
  • Return ErrorResponse with code, message, timestamp, traceId.
  • Map custom NotFoundException → 404, ValidationException → 400.
  • Same pattern in every microservice for client consistency.
SPRING 07

How does @Transactional work internally?

In simple words

Spring AOP proxy wraps your bean — opens DB transaction before method, commits on success, rolls back on RuntimeException.

What to tell the interviewer
  • Proxy via JDK/CGLIB — self-invocation bypasses transaction.
  • PlatformTransactionManager + connection bound to thread.
  • rollbackFor for checked exceptions when needed.
➤ Kafka & Messaging
KAFKA 01

Why Kafka instead of RabbitMQ?

In simple words

Kafka = durable replayable log, massive throughput, event streaming. RabbitMQ = traditional queue, routing, task delivery.

What to tell the interviewer
  • Choose Kafka: event sourcing, audit trail, replay, 1M+ events/day.
  • Choose RabbitMQ: simple task queues, complex routing, lower ops overhead.
KAFKA 02

How do partitions work?

In simple words

Topic split into partitions — parallel lanes. Each partition ordered. Producer picks partition by key hash — same key → same partition → ordering per key.

What to tell the interviewer
  • More partitions = more parallelism (one consumer per partition per group).
  • Partition key = orderId keeps all events for one order in order.
  • Replicas per partition for durability across brokers.
KAFKA 03

What happens if a consumer crashes?

In simple words

Kafka keeps messages on disk. Consumer group rebalances — another consumer takes over partitions. Uncommitted offset messages replayed.

What to tell the interviewer
  • Auto-commit vs manual commit — manual safer for at-least-once.
  • Idempotent consumer handles duplicate replay.
  • session.timeout.ms triggers rebalance when consumer dead.
KAFKA 04

How do you maintain message ordering?

In simple words

Order guaranteed only within one partition. Use same partition key (orderId) for related events. Single consumer per partition preserves order.

What to tell the interviewer
  • Don't use multiple threads on same partition without care.
  • Global order across topic = 1 partition (limits throughput).
  • Compromise: order per entity via key-based partitioning.
KAFKA 05

What is idempotency in distributed systems?

In simple words

Same request processed twice = same result as once. Critical when Kafka/network retries duplicate messages.

What to tell the interviewer
  • Idempotency key on payment API — store key + result in DB.
  • Consumer checks eventId already processed before side effect.
  • Kafka idempotent producer + transactional consumer patterns.
if (processedEvents.exists(event.getEventId())) return; // skip duplicate
processPayment(event);
processedEvents.save(event.getEventId());
➤ Database & JPA
JPA 01

First-level vs Second-level cache?

In simple words

L1 = session-scoped cache (Persistence Context) — automatic, per transaction. L2 = shared cache across sessions — Redis/Ehcache, optional.

What to tell the interviewer
  • L1: entity loaded once per EntityManager session.
  • L2: reference data (country codes) cached cluster-wide.
  • Invalidate L2 on writes; L1 cleared when session closes.
JPA 02

What is N+1 problem?

In simple words

1 query loads N parents, then N queries load each child — 101 queries instead of 1-2.

What to tell the interviewer
  • Fix: JOIN FETCH, @EntityGraph, batch fetching, DTO projection.
  • Detect: Hibernate statistics, p6spy, APM SQL count per request.
JPA 03

How to optimize slow queries?

In simple words

EXPLAIN the query, add indexes, paginate, avoid SELECT *, fix N+1, use read-only transactions for reports.

What to tell the interviewer
  • Index FK and WHERE columns; composite indexes for multi-column filters.
  • Keyset pagination instead of OFFSET on large tables.
  • Materialized views for heavy analytics.
JPA 04

How do you handle concurrent updates?

In simple words

Optimistic locking (@Version) — detect conflict on save. Pessimistic (SELECT FOR UPDATE) — lock row upfront. Retry on OptimisticLockException.

What to tell the interviewer
  • Optimistic: high read, low collision — e-commerce product update.
  • Pessimistic: inventory deduction, wallet balance — must not oversell.
  • Combine with idempotency for retry-safe APIs.
@Entity
class Account {
    @Version private Long version;
    private BigDecimal balance;
}
➤ System Design
DESIGN 01

Design a Payment / NEFT Processing System

In simple words

API receives payment → validate → idempotency check → persist PENDING → publish event → async settlement with bank → update SUCCESS/FAILED → notify customer.

What to tell the interviewer
  • Services: Payment, Account, Notification, Reconciliation batch.
  • Idempotency key + unique transaction ref (UTR for NEFT).
  • State machine: INITIATED → PROCESSING → SETTLED / FAILED.
  • Outbox + Kafka for reliable events; Saga for rollback on failure.
  • Reconciliation job matches bank statement vs ledger daily.
DESIGN 02

How to handle 1M+ transactions daily?

In simple words

~12 TPS average, higher peaks — horizontal scale, async Kafka, DB sharding/read replicas, caching, idempotent APIs.

What to tell the interviewer
  • Async pipeline — API returns 202, processing via Kafka consumers.
  • Partition Kafka by accountId; scale consumer pods.
  • DB: partition by date, archive old data, connection pooling.
  • Rate limit + queue during spikes; CDN not relevant for payments API.
DESIGN 03

Saga Pattern vs 2PC?

In simple words

2PC = all services commit or all rollback together — slow, tight coupling. Saga = local commits + compensating steps on failure — eventual consistency.

What to tell the interviewer
  • Microservices use Saga (choreography or orchestration).
  • 2PC/XA rare — doesn't scale across heterogeneous DBs.
  • Compensate: PaymentFailed → release inventory, mark order cancelled.
DESIGN 04

How to ensure data consistency across services?

In simple words

Database per service + eventual consistency via events. Outbox pattern, Saga, idempotency — not distributed JOINs.

What to tell the interviewer
  • Outbox: same TX saves business row + event row → relay to Kafka.
  • CDC (Debezium) alternative for event publishing.
  • Read your own writes: correlation ID + status polling or WebSocket.
DESIGN 05

How to implement distributed locking?

In simple words

When multiple instances need exclusive access — Redis Redisson lock, ZooKeeper, or DB advisory lock.

What to tell the interviewer
  • Redis: SET key NX EX ttl — fast, TTL prevents dead lock if node dies.
  • Use for scheduled job "only one pod runs reconciliation".
  • Prefer DB row lock for financial inventory when already in DB TX.
  • Always set lock timeout; avoid long-held distributed locks.
// Redisson
RLock lock = redisson.getLock("reconcile-daily");
if (lock.tryLock(0, 30, TimeUnit.MINUTES)) {
    try { runReconciliation(); }
    finally { lock.unlock(); }
}
➤ Production & DevOps
DEVOPS 01

How do you deploy microservices using Docker?

In simple words

Build fat JAR → Dockerfile multi-stage → push image to registry → Kubernetes/Docker Compose deploys containers with env config.

What to tell the interviewer
  • One image per microservice; tag with git SHA.
  • 12-factor: config via env vars / K8s ConfigMap.
  • Health checks: Actuator /actuator/health for readiness probe.
  • Rolling update — zero downtime deploy.
DEVOPS 02

What is Circuit Breaker?

In simple words

Stop calling failing service — fail fast with fallback until it recovers. Closed → Open → Half-Open.

What to tell the interviewer
  • Resilience4j @CircuitBreaker on WebClient/RestTemplate calls.
  • Prevents thread pool exhaustion when payment gateway is down.
  • Combine with timeout and retry (only idempotent ops).
DEVOPS 03

How do you monitor logs?

In simple words

Structured JSON logs with correlationId → centralized ELK/Splunk/Loki → search by traceId across all microservices.

What to tell the interviewer
  • MDC.put("correlationId", id) in gateway filter — propagates to all logs.
  • Promtail/Fluentd ship container logs to Loki/Elasticsearch.
  • Alert on error rate spike; link logs + metrics + traces (Grafana stack).
DEVOPS 04

What is Rate Limiting?

In simple words

Cap requests per client/IP — token bucket or sliding window in API Gateway or Redis — protects backend from abuse and spikes.

What to tell the interviewer
  • Gateway: 100 req/min per API key → HTTP 429 Too Many Requests.
  • Redis INCR + TTL for distributed rate limit across pods.
  • Different tiers: free vs premium customer limits.
2026 prep tip (3–5 YOE): Answer every question with architecture context — "In our payment microservice we used Saga + outbox because…" Interviewers expect production awareness, not textbook-only answers.
