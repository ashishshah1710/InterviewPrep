Spring Boot

Spring Framework fundamentals plus a deep-dive into internals — auto-configuration, startup flow, embedded servers, profiles, Actuator, and production pitfalls.

What does @SpringBootApplication do internally?

Annotation Bootstrap

@SpringBootApplication is a convenience composite annotation that combines three annotations:

@SpringBootConfiguration   // Specialized @Configuration
@EnableAutoConfiguration   // Enables Spring Boot auto-config
@ComponentScan             // Scans current package and sub-packages

When you run SpringApplication.run(App.class, args):

  • Creates an ApplicationContext (IoC container)
  • Registers all @Component, @Service, @Repository, @Controller beans via component scanning
  • Triggers auto-configuration based on classpath and properties
  • Starts the embedded web server (Tomcat/Jetty/Undertow) if web starter is present
  • Publishes ApplicationReadyEvent when fully initialized

What is @EnableAutoConfiguration, and how does it work?

Auto-Config

@EnableAutoConfiguration tells Spring Boot to automatically configure beans based on the jars present on the classpath. The mechanism:

  • Reads auto-config class names from META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
  • Each auto-config class is annotated with conditional annotations like @ConditionalOnClass, @ConditionalOnMissingBean, @ConditionalOnProperty
  • Only configurations whose conditions match are activated
  • Auto-configs are processed after user-defined beans, so your custom beans take precedence
Example: If spring-boot-starter-data-jpa is on the classpath and no DataSource bean exists, Spring Boot auto-configures a DataSource, EntityManagerFactory, and JpaTransactionManager using properties from application.properties.

What does @PostConstruct do, and at what stage is it executed?

Lifecycle

@PostConstruct (JSR-250) marks a method to run after the bean's dependency injection is complete but before the bean is put into service. It is part of the Spring bean lifecycle:

1. Instantiate bean (call constructor) 2. Populate properties (@Autowired / @Value) 3. BeanNameAware / BeanFactoryAware callbacks 4. @PostConstruct method executes ← HERE 5. InitializingBean.afterPropertiesSet() 6. Custom init-method (if configured) 7. Bean is ready for use --- On shutdown: 8. @PreDestroy method 9. DisposableBean.destroy()

Use @PostConstruct for initialization logic that depends on injected dependencies — loading caches, validating configuration, opening connections, or setting up internal state.

How does Dependency Injection work internally in Spring Boot?

IoC Container

Spring's DI is built on the Inversion of Control (IoC) principle — the framework creates and wires objects instead of your code calling new.

Internal flow:

  • Component scanning: Classpath is scanned for classes annotated with stereotype annotations. Each becomes a BeanDefinition.
  • Bean registration: Definitions are stored in the BeanFactory / ApplicationContext.
  • Instantiation: During context refresh, beans are created. Singletons are created once; prototypes per injection.
  • Dependency resolution: Spring analyzes constructor parameters, @Autowired fields/setters, and resolves matching beans by type (and @Qualifier by name).
  • Injection: Dependencies are injected via reflection (fields/setters) or constructor arguments.
  • Lifecycle callbacks: @PostConstruct, InitializingBean, etc. are invoked.
  • Proxy creation: For @Transactional, @Async, and AOP — Spring creates JDK dynamic proxies or CGLIB subclass proxies.

What is the difference between @Component, @Service, and @Repository?

Stereotypes

All three are specializations of @Component. They are functionally identical for Spring's component scanning and bean registration — Spring treats them the same internally.

How do you implement global exception handling in Spring Boot?

Exception Handling

Use @ControllerAdvice (or @RestControllerAdvice for REST APIs) combined with @ExceptionHandler to centralize exception handling across all controllers.

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(
            ResourceNotFoundException ex) {
        ErrorResponse error = new ErrorResponse(
            HttpStatus.NOT_FOUND.value(),
            ex.getMessage(),
            LocalDateTime.now()
        );
        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(error);
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ErrorResponse> handleValidation(
            MethodArgumentNotValidException ex) {
        String message = ex.getBindingResult().getFieldErrors().stream()
            .map(e -> e.getField() + ": " + e.getDefaultMessage())
            .collect(Collectors.joining(", "));
        return ResponseEntity.badRequest()
            .body(new ErrorResponse(400, message, LocalDateTime.now()));
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleGeneral(Exception ex) {
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR)
            .body(new ErrorResponse(500, "Internal server error",
                                    LocalDateTime.now()));
    }
}

Benefits: consistent error response format, no try-catch in every controller, separation of error handling from business logic, and proper HTTP status codes.

What happens if @Autowired is not used in Spring?

DI

Without Spring managing the object:

  • If you create an object with new MyService(), Spring does not inject dependencies — all @Autowired fields remain null.
  • Only classes registered as Spring beans (via stereotype annotations or @Bean methods) participate in dependency injection.
  • Exception: Since Spring 4.3, if a class has only one constructor, Spring implicitly autowires it even without @Autowired.
  • Manual wiring (setter calls) works but defeats the purpose of DI and makes testing harder.

Why do we use @Qualifier with @Autowired?

DI

When multiple beans implement the same type (or match the injection point), Spring cannot decide which one to inject. @Qualifier specifies the bean by name.

@Service("stripePayment")
class StripePaymentService implements PaymentService { }

@Service("paypalPayment")
class PayPalPaymentService implements PaymentService { }

@Service
class OrderService {
    public OrderService(
        @Qualifier("stripePayment") PaymentService paymentService) {
        // injects StripePaymentService bean
    }
}

Alternatives: @Primary on one bean (default when no qualifier), custom @Qualifier annotation (type-safe), or inject List<PaymentService> for all implementations.

What is Thread Scope?

Spring Scope

In Spring, thread scope means a bean instance is created once per thread and reused for all requests/work on that thread. Spring Framework 6.2+ provides built-in support via custom scope or community libraries; in practice it is often implemented with ThreadLocal or request-like context in worker threads.

// Conceptual: one UserContext per request thread
@Component
@Scope(value = WebApplicationContext.SCOPE_REQUEST, proxyMode = TARGET_CLASS)
public class RequestContext { }

// For async worker threads, use TaskDecorator to propagate context:
@Bean
public ThreadPoolTaskExecutor taskExecutor() {
    ThreadPoolTaskExecutor ex = new ThreadPoolTaskExecutor();
    ex.setTaskDecorator(runnable -> () -> {
        try {
            // copy MDC / SecurityContext / ThreadLocal into worker thread
            runnable.run();
        } finally {
            ContextHolder.clear();
        }
    });
    return ex;
}

Common scopes for comparison: singleton (one per container), prototype (new instance every injection), request (one per HTTP request), session (one per HTTP session). Thread scope is useful for background jobs where each worker thread needs isolated state — always clean up ThreadLocal to avoid leaks.

Spring Boot Deep Dive — Real Interview Questions

Spring Boot looks easy until the interviewer goes deep. These 30 questions cover internal mechanics, startup flow, auto-configuration, and production pitfalls.

How does Spring Boot decide which auto-configuration to apply?

Auto-Config Internals

Spring Boot uses a conditional evaluation pipeline during context refresh:

  • AutoConfigurationImportSelector loads candidate classes from META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports on the classpath (Boot 3.x; Boot 2.x used spring.factories).
  • Candidates are filtered and sorted (order matters — e.g., @AutoConfigureBefore / @AutoConfigureAfter).
  • Each auto-config class is wrapped in configuration-class processing and evaluated against @Conditional annotations:
ConditionTypical check
@ConditionalOnClassIs a library on the classpath? (e.g., javax.sql.DataSource)
@ConditionalOnMissingBeanDid the user already define this bean?
@ConditionalOnPropertyIs spring.datasource.url set?
@ConditionalOnWebApplicationIs this a servlet/reactive web app?
@ConditionalOnResourceDoes a config file exist?

Only configs whose all conditions pass are registered. User-defined @Configuration and @Bean methods always win over auto-config when both could create the same bean (@ConditionalOnMissingBean backs off).

What happens internally when you add spring-boot-starter-web?

Starter

A starter is a dependency aggregator — it pulls in transitive jars and triggers matching auto-configurations. spring-boot-starter-web brings:

  • spring-web, spring-webmvc — MVC framework
  • spring-boot-starter-tomcat — embedded Tomcat (default servlet container)
  • spring-boot-starter-json — Jackson for JSON serialization
  • spring-boot-starter-logging — Logback by default

Auto-configurations activated include: DispatcherServletAutoConfiguration, ServletWebServerFactoryAutoConfiguration (Tomcat), HttpEncodingAutoConfiguration, JacksonAutoConfiguration, WebMvcAutoConfiguration, and error handling (ErrorMvcAutoConfiguration).

Result: an embedded Tomcat starts on port 8080, a DispatcherServlet is registered at /, and your @RestController beans are mapped to HTTP endpoints — zero XML, zero manual servlet setup.

Why does Spring Boot prefer convention over configuration?

Philosophy

Convention over configuration means sensible defaults cover 80% of use cases so developers focus on business logic, not infrastructure wiring.

  • Reduced boilerplate: No manual web.xml, DataSource XML, or component-scan XML.
  • Faster onboarding: Same project structure, same config file names, same starter patterns across teams.
  • Sensible defaults: Port 8080, HikariCP pool, Jackson JSON, Logback logging — all work out of the box.
  • Override when needed: Conventions are not locked in — properties, @Bean, and exclusions let you customize without fighting the framework.

Spring Boot encodes conventions as auto-configuration classes + default property values in application.properties inside each starter jar.

How does Spring Boot load application.properties internally?

Configuration

Property loading is handled by the Environment abstraction, built during application startup:

1SpringApplication creates StandardServletEnvironment (or reactive equivalent).
2ConfigDataEnvironmentPostProcessor loads config data from classpath, file system, and optional cloud config sources.
3Default search order: application.properties / application.yml in classpath:/, classpath:/config/, file ./, ./config/.
4Profile-specific files loaded: application-{profile}.properties (e.g., application-prod.yml).
5Properties bind to @ConfigurationProperties beans via ConfigurationPropertiesBindingPostProcessor.
6@Value("${key}") resolves placeholders via PropertySourcesPropertyResolver.

Later sources override earlier ones in the property source chain. Command-line args and OS env vars sit at high priority.

What is the exact startup flow of a Spring Boot application?

Startup
main() → SpringApplication.run() ├─ Create SpringApplication instance ├─ Deduce application type (Servlet / Reactive / None) ├─ Load ApplicationContextInitializers (SpringFactoriesLoader) ├─ Load ApplicationListeners ├─ Prepare Environment (properties, profiles, config data) ├─ Print banner (optional) ├─ Create ApplicationContext │ ├─ Servlet: AnnotationConfigServletWebServerApplicationContext │ └─ Register @SpringBootApplication as source ├─ prepareContext() — post-process, apply initializers ├─ refreshContext() — THE BIG ONE: │ ├─ Parse @Configuration classes │ ├─ Component scan → register bean definitions │ ├─ Process @Import → auto-configuration classes │ ├─ Instantiate singleton beans (DI wiring) │ ├─ @PostConstruct, InitializingBean callbacks │ └─ Start embedded web server (WebServerStartStopLifecycle) ├─ callRunners() — CommandLineRunner / ApplicationRunner └─ Publish ApplicationReadyEvent → app is live

Failure at any step during refresh() triggers FailureAnalyzer reports (e.g., "Failed to configure DataSource").

How does Spring Boot detect embedded Tomcat and configure it?

Embedded Server
  • ServletWebServerFactoryAutoConfiguration activates when servlet web classes are on classpath and no custom ServletWebServerFactory bean exists.
  • @ConditionalOnClass({ Servlet.class, Tomcat.class }) checks Tomcat is available (from spring-boot-starter-tomcat).
  • Creates a TomcatServletWebServerFactory bean configured via ServerProperties (port, context-path, SSL, compression).
  • During context refresh, ServletWebServerApplicationContext calls factory.getWebServer() which builds and starts Tomcat programmatically — no external installation.
  • DispatcherServlet is registered as a servlet on the embedded container.

Switch to Jetty/Undertow by excluding Tomcat and adding spring-boot-starter-jetty — same auto-config mechanism picks the available factory.

What happens if two beans of same type exist and no @Qualifier?

DI

Spring's resolution order:

  • If one bean is marked @Primary → that bean is injected.
  • If parameter/field name matches a bean name (e.g., stripePayment) → inject by name.
  • If neither applies → startup fails with NoUniqueBeanDefinitionException: "expected single matching bean but found 2".
Application won't start — this is fail-fast by design. Fix with @Qualifier, @Primary, or refactor to a single implementation / strategy pattern with explicit selection.

How does Spring Boot handle profile-specific configuration?

Profiles 
# application.yml (default)
server.port: 8080

# application-dev.yml
spring.datasource.url: jdbc:postgresql://localhost/dev

# application-prod.yml
spring.datasource.url: jdbc:postgresql://prod-host/prod

Activate profiles via spring.profiles.active=dev,prod (property, env var SPRING_PROFILES_ACTIVE, or command line --spring.profiles.active=prod).

  • Profile-specific property files merge into the Environment — profile properties override defaults.
  • @Profile("dev") on @Configuration or @Bean — bean only registered when profile is active.
  • @ConditionalOnProfile used internally in auto-config classes.

What is the role of SpringFactoriesLoader?

SPI

SpringFactoriesLoader loads implementation classes listed in META-INF/spring.factories (Spring Framework SPI mechanism). Spring Boot uses it to discover:

  • ApplicationContextInitializer — run before context refresh
  • ApplicationListener — early event listeners
  • FailureAnalyzer — human-readable startup error messages
  • EnvironmentPostProcessor — modify Environment before context creation
  • SpringApplicationRunListener — hooks into run lifecycle

Note (Boot 3): Auto-configuration classes moved to AutoConfiguration.imports, but SpringFactoriesLoader is still used for the extension points above. It's the plugin discovery mechanism — libraries register hooks without modifying Spring Boot core code.

How does Spring Boot reduce XML configuration completely?

Configuration

XML is replaced by three mechanisms working together:

  • Annotation-driven config: @Component, @Autowired, @Configuration, @Bean replace <bean> tags.
  • Auto-configuration: Replaces hundreds of lines of infrastructure XML (DataSource, transaction manager, MVC) with conditional Java config classes.
  • Externalized properties: application.yml replaces property placeholders in XML.
  • Component scanning: Replaces <context:component-scan>.

XML is still supported (@ImportResource) but virtually unused in modern Spring Boot projects.

How does Spring Boot manage dependency versions automatically?

BOM

Via the Bill of Materials (BOM) — spring-boot-dependencies POM imported in your build:

// Maven — spring-boot-starter-parent imports the BOM
<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.2.0</version>
</parent>

// Gradle — dependency management plugin
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-web'
    // no version needed — BOM pins it
}

The BOM defines tested, compatible versions for 200+ libraries (Spring Framework, Hibernate, Jackson, Tomcat, Logback, etc.). You override individual versions only when needed. This eliminates dependency hell and classpath conflicts.

What is the lifecycle of a Spring Bean in Spring Boot?

Lifecycle
1. BeanDefinition registered (scan / @Bean / auto-config) 2. Instantiation (constructor call) 3. Dependency injection (@Autowired, constructor args) 4. Aware interfaces (BeanNameAware, ApplicationContextAware) 5. BeanPostProcessor.beforeInitialization() 6. @PostConstruct 7. InitializingBean.afterPropertiesSet() 8. Custom init-method 9. How does Spring Boot handle externalized configuration?

Externalized Config

Configuration is externalized so the same jar runs in dev, staging, and prod with different settings — no rebuild required.

  • Property files: application.yml, profile-specific files
  • Environment variables: SERVER_PORT=9090 → binds to server.port
  • Command-line args: --server.port=9090 (highest priority among defaults)
  • @ConfigurationProperties: Type-safe binding to POJOs
  • Cloud config: Spring Cloud Config Server for centralized config across microservices
  • Kubernetes ConfigMaps/Secrets: Mounted as env vars or files

Priority (low → high): default properties in jar → application.properties → profile files → env vars → command line.

What happens if application.yml and application.properties both exist?

Configuration

Both are loaded into the Environment as separate property sources. If the same key exists in both, application.properties takes precedence over application.yml (within the same location/priority level).

Best practice: pick one format per project — mixing causes confusion. Use YAML for nested/hierarchical config; use properties for simple flat key-value setups.

Profile files follow the same rule: application-prod.properties overrides duplicate keys in application-prod.yml.

How does Spring Boot integrate with Actuator internally?

Actuator

Adding spring-boot-starter-actuator triggers EndpointAutoConfiguration and related auto-configs:

  • Registers endpoints as Spring beans (HealthEndpoint, MetricsEndpoint, InfoEndpoint, etc.) annotated with @Endpoint.
  • WebEndpointAutoConfiguration exposes endpoints over HTTP (or JMX) via @WebEndpoint / @ControllerEndpoint.
  • HealthContributor beans aggregate into /actuator/health (DB, disk, custom checks).
  • Micrometer integration auto-configures metrics export (Prometheus, Datadog).
  • Security auto-config can restrict endpoints — only /health and /info public by default in recent versions.

Actuator endpoints are discovered via classpath scanning + conditional auto-config — same pattern as everything else in Spring Boot.

How does Spring Boot create DataSource automatically?

DataSource

DataSourceAutoConfiguration activates when:

  • javax.sql.DataSource is on classpath (from JDBC starter or JPA starter)
  • No existing DataSource bean (@ConditionalOnMissingBean)
  • spring.datasource.url is set (or embedded DB like H2 detected)

It reads spring.datasource.* properties, creates a HikariDataSource (default pool), and registers it. If JPA starter is present, HibernateJpaAutoConfiguration chains off the DataSource to create EntityManagerFactory and JpaTransactionManager.

spring.datasource.url=jdbc:postgresql://localhost:5432/mydb
spring.datasource.username=app
spring.datasource.password=secret
spring.datasource.driver-class-name=org.postgresql.Driver
# HikariCP pool settings optional — sensible defaults apply

What is the real use of CommandLineRunner?

Startup Hook

CommandLineRunner (and ApplicationRunner) run after the ApplicationContext is fully started — the app is live, all beans are wired, embedded server is running.

@Component
public class DataSeeder implements CommandLineRunner {
    private final UserRepository repo;
    public DataSeeder(UserRepository repo) { this.repo = repo; }

    @Override
    public void run(String... args) {
        if (repo.count() == 0) {
            repo.saveAll(defaultUsers());
        }
    }
}

Real uses: seed database, warm caches, print startup diagnostics, run one-time migrations, load reference data, validate external service connectivity. Ordered via @Order or Ordered interface when multiple runners exist.

How does Spring Boot handle exception translation?

Exceptions
  • @Repository: PersistenceExceptionTranslationPostProcessor wraps beans in a proxy that converts vendor exceptions (e.g., SQLException, Hibernate exceptions) into Spring's unchecked DataAccessException hierarchy.
  • Web layer: DispatcherServlet catches exceptions and delegates to HandlerExceptionResolver chain — including your @ControllerAdvice methods.
  • Validation: MethodArgumentNotValidException from @Valid → handled by exception resolver → 400 response.
  • Default error handling: BasicErrorController serves /error with JSON error payload (timestamp, status, message).

How does Spring Boot support microservices so easily?

Microservices
  • Standalone executable jars: Each service is independently deployable.
  • Opinionated defaults: REST APIs, JSON, health checks ready in minutes.
  • Spring Cloud integration: Service discovery (Eureka/Consul), config server, API gateway, circuit breakers plug in via starters.
  • Actuator: Health, metrics, info endpoints for Kubernetes liveness/readiness probes.
  • Externalized config + profiles: Same artifact, different config per environment/service.
  • Embedded server: No shared app server — true process isolation.

How does Spring Boot handle logging by default?

Logging
  • spring-boot-starter-logging brings Logback by default (Log4j2 if you exclude Logback and add log4j starter).
  • Default config in spring-boot-{version}.jar → base.xml: console appender, color output, log level INFO for root, DEBUG for your app package if dev profile.
  • Override via src/main/resources/logback-spring.xml or properties: logging.level.com.myapp=DEBUG, logging.file.name=app.log.
  • All logging frameworks bridged through SLF4J — no direct Log4j/Logback calls needed in code.

How does Spring Boot decide server port priority?

Server

Port resolution follows the standard property precedence chain. Highest wins:

Priority (high → low)Source

2OS environment variable: SERVER_PORT=9090

Why is Spring Boot preferred for cloud-native apps?

Cloud Native
  • 12-factor app alignment: Config in environment, stateless processes, port binding, disposability (graceful shutdown).
  • Container-friendly: Single fat jar, fast startup (especially with GraalVM native images), low memory tuning.
  • Kubernetes-ready: Actuator health probes, Prometheus metrics, externalized config via ConfigMaps.
  • Microservice ecosystem: Spring Cloud for discovery, config, resilience, gateway.
  • Observability: Micrometer + OpenTelemetry integration out of the box.
