1-Minute Java Multithreading Cheat Sheet

Every Java developer should know these — quick revision before an interview. Based on the core multithreading roadmap.

⏱ 1 MINUTE REVISION

Java Multithreading — Every Developer Should Know

Eight core topics + thread methods + best practices + interview checklist.

01 · WHY IT MATTERS

What is Multithreading?

In simple words

CPU runs multiple parts of your program at once (threads). Better CPU use — while one thread waits on network, another works.

What to tell the interviewer
  • Improves throughput and responsiveness for I/O-heavy apps.
  • Example: browser downloads file in background while you scroll — two threads.
  • Every Java app starts with at least one thread: the main thread.
  • Tomcat creates one thread per HTTP request — multithreading by default in web apps.
02 · THREAD VS PROCESS

Thread vs Process

In simple words

Process = whole program (Chrome app). Thread = one task inside it (one tab). Threads share memory; processes don't.

What to tell the interviewer
  • Thread: lightweight, shared heap, fast switch, easy communication.
  • Process: isolated memory, heavy, safe crash isolation.
  • Draw: one Process box containing Thread 1, Thread 2, Thread 3.

03 · CREATING THREADS

Creating Threads

In simple words

Two classic ways: extend Thread, or implement Runnable and pass to Thread. Always call start() — not run().

What to tell the interviewer
  • Method 1: class MyThread extends Thread — rarely use (wastes inheritance).
  • Method 2: implements Runnable — preferred for separating task from thread.
  • Trap: run() = normal method call in current thread. start() = new thread.
  • Production: use ExecutorService instead of new Thread().
// Method 1
class MyThread extends Thread {
    public void run() { System.out.println("Running"); }
}
new MyThread().start();

// Method 2 (preferred)
class MyTask implements Runnable {
    public void run() { System.out.println("Task"); }
}
new Thread(new MyTask()).start();
04 · RUNNABLE

Runnable Interface

In simple words

Runnable defines the job (run method). Thread or thread pool runs the job — flexible and reusable.

What to tell the interviewer
  • Functional interface — use lambda: () -> doWork().
  • Same Runnable can run on different executors.
  • Need return value? Use Callable + Future instead.
05 · SYNCHRONIZATION

Synchronization

In simple words

Lock so only one thread updates shared data at a time — like one person editing a shared document at a time.

What to tell the interviewer
  • synchronized on method or block — mutual exclusion.
  • Sync only what you must — less scope = better performance.
  • Alternatives: ReentrantLock, AtomicInteger, ConcurrentHashMap.
class Counter {
    private int count = 0;
    public synchronized void increment() { count++; }
}
06 · RACE CONDITIONS

Race Conditions

In simple words

Two threads change the same variable at once — result depends on lucky timing. count++ without sync is the classic bug.

What to tell the interviewer
  • Fix: synchronization, locks, or Atomic classes.
  • Symptom: flaky tests, wrong totals under load.
  • Prefer AtomicInteger over synchronized for simple counters.
Thread 1 ──┐ ├──▶ Shared counter ──▶ Unpredictable result Thread 2 ──┘ Fix: lock or AtomicInteger
07 · EXECUTOR FRAMEWORK

Executor Framework (java.util.concurrent)

In simple words

Reuse a pool of threads instead of creating new ones each time — faster, controlled, production-ready.

What to tell the interviewer
  • newFixedThreadPool(n) — fixed workers, bounded.
  • newCachedThreadPool() — grows/shrinks, good for many short tasks.
  • newSingleThreadExecutor() — one at a time, ordered.
  • newScheduledThreadPool(n) — delayed/periodic jobs.
  • Modern Java: prefer Executor over manual new Thread().
ExecutorService pool = Executors.newFixedThreadPool(4);
pool.submit(() -> processOrder(orderId));
pool.shutdown();
08 · PROJECTS

Projects That Prove You Understand Multithreading

In simple words

Build something real — thread pool downloader, seat booking with race fix, Spring @Async order emails.

What to tell the interviewer
  • Producer-consumer with BlockingQueue.
  • Parallel URL fetcher with ExecutorService.
  • Demonstrate race, then fix with AtomicInteger — shows depth.

BONUS · BEST PRACTICES

Multithreading Best Practices

In simple words

Use pools, minimize locks, avoid deadlocks, pick thread-safe collections from java.util.concurrent.

What to tell the interviewer
  • Don't create unlimited threads — use a pool with bounds.
  • Minimize synchronized scope — avoid blocking I/O inside locks.
  • Avoid deadlock: consistent lock ordering, tryLock with timeout.
  • Use ConcurrentHashMap, CopyOnWriteArrayList — not synchronized wrappers.
  • Always shutdown ExecutorService — or JVM won't exit cleanly.
INTERVIEW CHECKLIST

Common Multithreading Interview Questions — Can you answer these?

In simple words

Revise these six before any Java interview — answer with when/why + project example, not definition only.

  • What is multithreading and why is it useful?
  • Difference between Thread class and Runnable interface?
  • Difference between start() and run()?
  • How does synchronization work?
  • What is a race condition and how do you fix it?
  • What is the Executor Framework and why use it?
Pro tip: For each question, answer: definition → when you used it in a project → what goes wrong without it. That separates beginners from professionals.
