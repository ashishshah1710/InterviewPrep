Microservices Event-Driven Architecture

End-to-end workflow — from client request through API Gateway, Kafka events, downstream services, and polyglot persistence. Enterprise-grade interview explanation.

Event-Driven Microservices — Complete Workflow

Services communicate asynchronously via events while staying independent, scalable, and deployable on their own.

ARCHITECTURE OVERVIEW

End-to-End Event-Driven Workflow

In simple words

Client places order through the gateway. Order Service saves the order and publishes an event to Kafka. Inventory, Payment, Shipping, and Notification services listen independently and react — no direct calls between them.

What to tell the interviewer
  • Start with the user journey, then zoom into async event flow.
  • Emphasize: each service owns its DB — no shared database anti-pattern.
  • Order Service returns 202 Accepted quickly — processing continues async.
  • Mention idempotency, outbox pattern, and dead-letter queue for production.
  • Draw Client → Gateway → Order → Kafka → 4 consumers on whiteboard.

Architecture benefits:

Loose coupling — services don't call each other directly
High scalability — scale consumers independently
Fault tolerance — Kafka retains events; consumers retry
Real-time processing — events processed as they arrive
Independent deployment — deploy Payment without touching Order
System resilience — one service down doesn't block others
STEP 01 · CLIENT

Client Request

In simple words

User submits an order from web or mobile app — JSON payload with items, address, payment method. Hits the API Gateway, never microservices directly.

What to tell the interviewer
  • POST /api/orders with JWT in Authorization header.
  • Client gets orderId + status PENDING immediately (async model).
  • Poll GET /orders/{id} or use WebSocket/SSE for status updates.
POST /api/orders
Authorization: Bearer <jwt>

  "customerId": "C-101",
  "items": [{ "productId": "P-42", "qty": 2 }],
  "shippingAddress": { ... },
  "paymentMethod": "CARD"
}
STEP 02 · GATEWAY

API Gateway — Authentication, Authorization, Rate Limiting

In simple words

Single front door validates JWT, checks user role, limits requests per minute, routes to Order Service, adds correlation ID for tracing.

What to tell the interviewer
  • Auth: validate JWT signature, expiry, issuer (Keycloak/OAuth2).
  • Authorization: role CUSTOMER vs ADMIN on route level.
  • Rate limiting: Redis token bucket — 100 req/min per client.
  • Tools: Spring Cloud Gateway, Kong, AWS API Gateway.
  • Inject X-Correlation-Id for distributed tracing across all services.
STEP 03 · ORDER SERVICE

Order Service Processing

In simple words

Validates order, saves to its own DB with status CREATED, publishes OrderCreated event, returns 202 Accepted — doesn't wait for payment or shipping.

What to tell the interviewer
  • Owns orders table — single source of truth for order state.
  • State machine: CREATED → PAID → SHIPPED → DELIVERED (or FAILED).
  • Use Outbox pattern: save order + event in same DB transaction.
  • Separate process reads outbox and publishes to Kafka — guaranteed delivery.
@Transactional
public OrderResponse createOrder(CreateOrderRequest req) {
    Order order = orderRepo.save(new Order(req, Status.CREATED));
    outboxRepo.save(new OutboxEvent("OrderCreated", order.toEvent()));
    return new OrderResponse(order.getId(), Status.CREATED); // 202 Accepted
}
STEP 04 · EVENT PUBLISHING

Event Publishing

In simple words

After order is saved, an event message is sent to Kafka — a JSON record saying "OrderCreated" with orderId, items, customerId. Other services subscribe to react.

What to tell the interviewer
  • Event schema: orderId, eventType, timestamp, payload, correlationId.
  • Use Avro/JSON Schema for contract evolution.
  • Partition key = orderId — all events for one order stay ordered.
  • Producer acks=all for durability; idempotent producer enabled.
{
  "eventId": "evt-uuid",
  "eventType": "OrderCreated",
  "orderId": "ORD-9001",
  "customerId": "C-101",
  "items": [{ "productId": "P-42", "qty": 2 }],
  "timestamp": "2026-06-22T10:30:00Z",
  "correlationId": "corr-abc123"
}
STEP 05 · EVENT BUS

Event Bus / Kafka Integration

In simple words

Kafka is the central message log. Producers append events; consumers read at their own pace. Messages persist and can be replayed if a service was down.

What to tell the interviewer
  • Topics: order.events, payment.events, inventory.events.
  • Consumer groups: inventory-group, payment-group — each gets every message once per group.
  • Retention: 7 days — replay for recovery or new consumer.
  • Dead Letter Topic (DLT) for failed messages after N retries.
  • Spring: KafkaTemplate publish, @KafkaListener consume.
@KafkaListener(topics = "order.events", groupId = "inventory-group")
public void onOrderCreated(OrderCreatedEvent event) {
    inventoryService.reserveStock(event.getOrderId(), event.getItems());
}
STEP 06 · DOWNSTREAM SERVICES

Inventory, Payment, Shipping & Notification Services

In simple words

Four independent services each listen to OrderCreated (or chained events) and do their job: reserve stock, charge card, create shipment, send email — in parallel where possible.

What to tell the interviewer
  • Inventory: reserve stock → publish StockReserved or StockFailed.
  • Payment: on StockReserved → charge → publish PaymentCompleted/Failed.
  • Shipping: on PaymentCompleted → create label → publish ShipmentCreated.
  • Notification: listens to all status events → email/SMS/push.
  • Each consumer idempotent — same event processed twice = same result.
  • Saga orchestration or choreography — explain which you used.
OrderCreated (Kafka) ├─▶ Inventory Service → StockReserved / StockFailed ├─▶ Payment Service → (waits StockReserved) → PaymentCompleted ├─▶ Notification → "Order received" email └─▶ Order Service → (listens back) updates status PaymentCompleted ├─▶ Shipping Service → ShipmentCreated └─▶ Notification → "Payment confirmed"
STEP 07 · DATABASES

Independent Databases — Polyglot Persistence

In simple words

Each microservice has its own database — no shared tables. Pick the best DB per service: PostgreSQL for transactions, MongoDB for flexible shipment docs, Redis for cache.

What to tell the interviewer
  • Database-per-service — core microservices principle.
  • Order → PostgreSQL (ACID). Inventory → PostgreSQL. Shipping → MongoDB. Cache → Redis.
  • No cross-DB joins — use API or events to get data from other services.
  • Eventual consistency accepted — order status catches up async.

STEP 08 · EVENT CHAINING

Event Chaining & Asynchronous Communication

In simple words

One event triggers the next — OrderCreated → StockReserved → PaymentCompleted → ShipmentCreated. Each step is async; failures trigger compensating events (Saga).

What to tell the interviewer
  • Choreography: services react to events without central coordinator.
  • Orchestration: Saga orchestrator sends commands — easier to trace.
  • Compensation: PaymentFailed → release stock, cancel order (Saga undo).
  • All async — no long synchronous chain blocking the user.
  • Observability: correlationId in every event → single trace in Jaeger.
// Saga compensation example
@KafkaListener(topics = "payment.events")
public void onPaymentFailed(PaymentFailedEvent e) {
    inventoryService.releaseStock(e.getOrderId());  // compensate
    orderService.markFailed(e.getOrderId());
    notificationService.sendPaymentFailedEmail(e);
}
INTERVIEW SCRIPT

How to explain this architecture in a 3-minute interview answer

In simple words

Tell the story: user orders → gateway secures request → order saved → Kafka event → four services react independently → each has own DB → failures handled with Saga and retries.

What to tell the interviewer
  • Minute 1: Business problem — scale order flow, decouple teams, survive partial failures.
  • Minute 2: Walk the flow — Gateway → Order → Kafka → parallel consumers.
  • Minute 3: Production concerns — outbox, idempotency, DLT, tracing, Saga compensation.
  • Close with benefit: "We deployed Payment service 12 times last sprint without touching Order service."
Enterprise context: In large-scale applications (banking, e-commerce, logistics), Event-Driven Architecture lets teams ship independently, handle peak load (Black Friday), and recover from partial outages without full system downtime.
