Common Interview Questions — Latency & Throughput

Performance fundamentals every backend engineer must explain clearly — bandwidth, caching, scaling, and real-world trade-offs.

PERFORMANCE · SYSTEM DESIGN

Latency vs Throughput vs Bandwidth

10 frequently asked questions — with simple explanations and interview talking points.

LatencyTime for one request to complete (ms). "How long does one call take?"
ThroughputHow many requests complete per second (req/s, Mbps). "How much work per unit time?"
Common Interview Questions
PERF 01

What is the difference between latency and throughput?

In simple words

Latency = delay for a single operation (e.g. API responds in 200 ms). Throughput = total work done over time (e.g. 5,000 requests/sec). A fast single request (low latency) doesn't automatically mean the system handles many requests (high throughput).

What to tell the interviewer
  • Latency measured in ms; throughput in ops/sec or Mbps.
  • Example: one DB query = 5 ms latency; connection pool with 100 threads might achieve 20,000 queries/sec throughput.
  • Often a trade-off: batching increases throughput but adds latency per item.
  • SLIs: p50/p99 latency for user experience; throughput for capacity planning.
// Low latency: respond immediately with cached data
// High throughput: process 10K Kafka messages/sec with parallel consumers
PERF 02

Can a system have high bandwidth but low throughput?

In simple words

Yes. Bandwidth is the maximum capacity of the link (pipe width). Throughput is what you actually use. A 1 Gbps link with a slow server, CPU bottleneck, or packet loss may only deliver 100 Mbps of real throughput.

What to tell the interviewer
  • Bandwidth = theoretical max; throughput = actual delivered data.
  • Bottlenecks: CPU-bound app, disk I/O, TCP retransmits, single-threaded processing.
  • Analogy: 8-lane highway (bandwidth) but accident causes traffic jam (low throughput).
  • Fix throughput by removing bottleneck — not always by upgrading the link.
PERF 03

Why can a network with 100 Mbps bandwidth achieve only 60 Mbps throughput?

In simple words

The link can carry 100 Mbps max, but real-world factors eat into it — protocol overhead, congestion, retransmissions, Wi-Fi interference, and limits on the sending/receiving devices.

What to tell the interviewer
  • TCP/IP headers + ACKs reduce usable payload (~5–15% overhead).
  • Packet loss → retransmits; congestion control throttles speed.
  • Shared link: other users/apps consume part of the 100 Mbps.
  • Server disk/CPU can't read/send fast enough; old NIC or Wi-Fi half-duplex.
  • Measure with iperf — compare to ISP rated speed to isolate layer.
PERF 04

How does caching impact latency?

In simple words

Cache hit = data from Redis/memory in ~1 ms instead of DB in ~50 ms — latency drops sharply. Cache miss = extra hop to cache + DB, so latency can be slightly worse unless cache is well-designed.

What to tell the interviewer
  • Cache hit: skip DB/network round trip — major latency win.
  • CDN edge cache: serve static assets closer to user — cuts geographic latency.
  • Cache-aside: app checks Redis → on miss, load DB and populate cache.
  • Stale cache TTL vs consistency trade-off; warm cache on deploy.
  • Project example: "Product API p99 dropped from 120 ms to 8 ms with Redis."
// Cache hit:  ~1 ms   (Redis local)
// Cache miss: ~55 ms  (Redis miss + PostgreSQL query + cache write)
// No cache:   ~50 ms  (PostgreSQL every time)
PERF 05

How does horizontal scaling impact throughput?

In simple words

Add more identical service instances behind a load balancer — each handles requests in parallel, so total throughput increases roughly linearly (until DB or Kafka becomes the bottleneck).

What to tell the interviewer
  • 3 pods × 2,000 req/s each ≈ 6,000 req/s system throughput.
  • Requires stateless services — session in Redis, not in-memory on one pod.
  • Doesn't fix slow single-request latency if code/DB query is the problem.
  • Downstream limits: DB connection pool, Kafka partition count cap parallelism.
  • K8s HPA scales on CPU or custom metric (queue depth, req/s).
PERF 06

What factors contribute to network latency?

In simple words

Every hop adds delay: distance (speed of light), router/firewall processing, packet size, queueing during congestion, DNS lookup, and TLS handshake.

What to tell the interviewer
  • Propagation: Mumbai → US East ≈ 200 ms RTT (physics).
  • Transmission: larger packets take longer on a given link speed.
  • Processing: load balancer, API Gateway, WAF inspect each packet.
  • Queuing: congestion at peak — packets wait in buffers.
  • Application: DNS (~20–100 ms), TLS 1.3 handshake (~1 RTT).
PERF 07

Why does increasing bandwidth not always reduce latency?

In simple words

Bandwidth and latency measure different things. Bandwidth = how much data fits at once (wider pipe). Latency = how long one bit takes to travel (pipe length). A wider pipe doesn't make light travel faster.

What to tell the interviewer
  • Upgrading 100 Mbps → 1 Gbps helps when link is saturated (throughput bound).
  • If latency is 200 ms due to geography, more bandwidth won't change RTT.
  • Serial request chains (A→B→C→D) add latencies — bandwidth doesn't parallelize that.
  • Fix latency: reduce hops, cache, CDN, async, co-locate services.
  • Fix throughput: more bandwidth, batching, horizontal scale, partitioning.
PERF 08

What matters more for online gaming: latency or throughput?

In simple words

Latency (ping). Games send tiny packets frequently — you need low RTT (<50 ms ideal) so actions register instantly. High Mbps download speed doesn't help if ping is 200 ms.

What to tell the interviewer
  • Throughput needs are low (KB/s) — latency is the bottleneck.
  • UDP often preferred over TCP for real-time (less retransmit delay).
  • Game servers deployed in regional zones close to players.
  • Jitter (latency variation) also matters for smooth gameplay.
PERF 09

What matters more for video streaming: latency or bandwidth?

In simple words

Bandwidth (mostly). HD needs ~5 Mbps, 4K needs ~25 Mbps sustained throughput. A few seconds of buffer absorbs latency — users care about quality, not instant frame delivery (except live sports).

What to tell the interviewer
  • On-demand Netflix: buffer 10–30 sec ahead — latency less critical.
  • Live streaming / video calls: latency becomes important again.
  • Adaptive bitrate switches quality based on available bandwidth.
  • CDN delivers high throughput by caching video segments at edge.
PERF 10

How would you reduce latency in a distributed system?

In simple words

Cut round trips: cache hot data, co-locate services, use connection pooling, CDN for static assets, async for non-critical paths, and deploy closer to users geographically.

What to tell the interviewer
  • Cache: Redis for reads; CDN for static/API at edge.
  • Fewer hops: merge microservices or use BFF to avoid chatty REST chains.
  • Async: return 202 + poll/WebSocket instead of sync wait on 3 services.
  • DB: indexes, read replicas, avoid N+1; connection pool tuning.
  • Network: gRPC + HTTP/2 multiplexing internally; keep-alive; regional deployment.
  • Observe: distributed tracing (Jaeger) to find slowest span in the chain.
// Before: Client → Gateway → Order → Inventory → Payment → DB  (4 hops, 400 ms)
// After:  Client → Gateway → Order (cache inventory, async payment)  (~80 ms)
//         + Redis cache hit on product catalog
