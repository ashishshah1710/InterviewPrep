How to Design a High-Throughput REST API (Millions of Requests)

System design question — show structured thinking, not a technology laundry list.

Sample 2-minute interview script

Talking Points
Step 1 — Architecture

"I'd design a horizontally scalable, stateless REST API. Traffic enters through a load balancer or API gateway for TLS, auth, and rate limiting. Stateless app instances run in an auto-scaling group."

Step 2 — Scale reads

"For millions of requests, most APIs are read-heavy. I'd add Redis cache-aside and optionally a CDN for cacheable GET endpoints to reduce DB load."

Step 3 — Database

"Primary for writes, read replicas for reads, with indexing and query optimization. Heavy work goes to an async queue."

Step 4 — Trade-off

"I'd avoid premature sharding. Scale vertically + replicas + cache first. Shard only when metrics prove we need it."

Follow-ups to expect: cache stampede (TTL jitter, request coalescing), hot keys (local cache, key splitting), strong consistency (read-your-writes via primary routing), write scaling (partition by tenant ID, CQRS).

API Gateway vs Load Balancer

Many engineers use these interchangeably — they aren't the same. In production you often need both.

The one-line difference

System Design
Load Balancer: Where should this request go? (which healthy instance)
API Gateway: What should happen to this request before it gets there? (policies, auth, routing)

Different responsibilities. Better together.

API Gateway — request & API policy layer

API Gateway

Acts as the single entry point for client requests at the application/API layer.

ResponsibilityWhat it does
Authentication & authorizationValidate JWT, API keys, OAuth tokens before traffic hits services
Rate limiting & throttlingProtect backends from abuse; return 429 with Retry-After
Request validationSchema validation, size limits, required headers
API versioningRoute /v1/users vs /v2/users to different backends
Protocol translationREST ↔ gRPC, HTTP ↔ WebSocket at the edge
RoutingSend request to the correct microservice (orders, payments, users)
ObservabilityCentral logging, metrics, tracing, request/response transformation

Focus: How a request should be processed before it reaches a service.

Examples: AWS API Gateway, Kong, Apigee, Spring Cloud Gateway, NGINX with auth plugins.

Load Balancer — traffic & availability layer

Load Balancer

Distributes incoming traffic across multiple instances of the same service for availability and scale.

ResponsibilityWhat it does
Traffic distributionRound-robin, least connections, IP hash across healthy nodes
Health checksStop sending traffic to failed/unhealthy instances
FailoverRoute around dead nodes automatically
SSL/TLS terminationDecrypt HTTPS at the edge; backends can use HTTP internally
High availabilitySpread load so no single server is overwhelmed

Focus: Which healthy instance should receive the request.

Examples: AWS ALB/NLB, NGINX, HAProxy, F5, Kubernetes Service + kube-proxy.

Typical microservices flow

Architecture 
Client
  → Load Balancer          (distributes across healthy gateway instances)
  → API Gateway            (auth, rate limit, validate, route by path/version)
  → Microservices          (User Service | Order Service | Payment Service)
       ↓
  Each microservice may have its own Load Balancer internally
  • The Load Balancer ensures traffic is evenly spread across healthy API Gateway instances.
  • The API Gateway authenticates, validates, applies policies, and routes to the correct backend service.
  • Inside each microservice cluster, another LB often distributes across app pods/instances.

Interview answer (30 seconds)

Talking Points
Script

"They're not interchangeable. A Load Balancer distributes traffic across multiple instances of the same service — it answers where the request should go based on health and load. An API Gateway is the application entry point — it handles auth, rate limiting, validation, versioning, and routes to the right microservice. It answers what should happen to the request before it reaches backend code. In production microservices, the typical flow is Client → Load Balancer → API Gateway → Services. You need both: LB for availability and scale at the infrastructure layer, Gateway for API policies at the application layer."

Common mistake in interviews: Saying "we use API Gateway for load balancing." Correct framing: the gateway routes to services; the load balancer distributes across instances. Some tools (e.g. NGINX, ALB) blur the line — acknowledge that, then explain the conceptual separation.

Find First Non-Repeating Character in String (Java 8)

Optimal approach — O(n) time, O(1) space (fixed alphabet)

DSA · Java 8

Two-pass with frequency array or HashMap. For ASCII/lowercase a–z, a int[26] or int[256] is O(1) space.

// Approach 1: LinkedHashMap — preserves insertion order, O(n) time
public static Character firstNonRepeating(String s) {
    if (s == null || s.isEmpty()) return null;

    Map<Character, Long> freq = s.chars()
        .mapToObj(c -> (char) c)
        .collect(Collectors.groupingBy(c -> c, LinkedHashMap::new, Collectors.counting()));

    return freq.entrySet().stream()
        .filter(e -> e.getValue() == 1)
        .map(Map.Entry::getKey)
        .findFirst()
        .orElse(null);
}

// Approach 2: Two-pass array — fastest for a-z, O(n) time, O(1) space
public static Character firstNonRepeatingFast(String s) {
    int[] count = new int[26];
    for (char c : s.toCharArray()) {
        count[c - 'a']++;
    }
    for (char c : s.toCharArray()) {
        if (count[c - 'a'] == 1) return c;
    }
    return null;
}

// Example: "swiss" → 'w'  |  "aabb" → null
Explain complexity
  • Time: O(n) — each character visited at most twice
  • Space: O(1) for fixed alphabet; O(k) for HashMap where k = unique chars
  • Why LinkedHashMap: HashMap doesn't guarantee order; we need first occurrence in string order

Find Duplicates in int Array (Minimal Time Complexity)

Approaches ranked by use case

DSA
ApproachTimeSpaceWhen to use
HashSetO(n)O(n)General case — find if duplicate exists or collect all dupes
Sorting + scanO(n log n)O(1)*When in-place sort allowed, memory constrained
Floyd's cycle (index as pointer)O(n)O(1)Array values in range [1..n], find one duplicate
Boolean array / bitsetO(n)O(n)Values in known small range [0..n]
// Find ALL duplicates — O(n) time, O(n) space
public static List<Integer> findDuplicates(int[] nums) {
    Set<Integer> seen = new HashSet<>();
    Set<Integer> duplicates = new LinkedHashSet<>(); // preserves order

    for (int num : nums) {
        if (!seen.add(num)) {  // add returns false if already present
            duplicates.add(num);
        }
    }
    return new ArrayList<>(duplicates);
}

// Java 8 stream variant — check if ANY duplicate exists
boolean hasDuplicate = Arrays.stream(nums).boxed()
    .collect(Collectors.groupingBy(i -> i, Collectors.counting()))
    .values().stream()
    .anyMatch(count -> count > 1);

// Find ONE duplicate when values are in [1..n] and array size n+1 — Floyd O(n), O(1)
public static int findDuplicateFloyd(int[] nums) {
    int slow = nums[0], fast = nums[0];
    do {
        slow = nums[slow];
        fast = nums[nums[fast]];
    } while (slow != fast);
    slow = nums[0];
    while (slow != fast) {
        slow = nums[slow];
        fast = nums[fast];
    }
    return slow;
}

How to Select Top 10 Rows from Database

Syntax by database + real use cases

SQL 
-- Standard SQL (PostgreSQL, SQL Server 2012+, Oracle 12c+)
SELECT * FROM employees
ORDER BY salary DESC
FETCH FIRST 10 ROWS ONLY;

-- MySQL / PostgreSQL shorthand
SELECT * FROM employees
ORDER BY salary DESC
LIMIT 10;

-- Oracle (legacy)
SELECT * FROM (
    SELECT e.*, ROWNUM rn FROM employees e ORDER BY salary DESC
) WHERE rn <= 10;

-- SQL Server (legacy)
SELECT TOP 10 * FROM employees ORDER BY salary DESC;

-- Top 10 PER department (window function — very common interview follow-up)
SELECT * FROM (
    SELECT *,
           ROW_NUMBER() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rn
    FROM employees
) ranked
WHERE rn <= 10;
Performance note: ORDER BY ... LIMIT 10 is fast only if there's an index supporting the sort column. Without an index, DB must sort the entire table. Mention this in interviews.

What Is Lazy Loading?

Concept + Hibernate/JPA example

JPA / Hibernate

Lazy loading defers loading of related data until it is actually accessed. Eager loading loads everything upfront in one query.

@Entity
public class Order {
    @Id
    private Long id;

    // LAZY (default for @OneToMany, @ManyToMany):
    // OrderItems are NOT loaded when you fetch Order
    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
    private List<OrderItem> items = new ArrayList<>();
}

// Usage — items loaded only when accessed (inside active session)
Order order = orderRepo.findById(1L);   // SELECT order only
order.getItems().size();                 // SELECT items NOW (lazy trigger)

// EAGER — loads order + items in one or multiple queries immediately
@ManyToOne(fetch = FetchType.EAGER)
private Customer customer;
LazyInitializationException: Accessing lazy collection outside a transaction/session throws this. Fix with @Transactional on service method, JOIN FETCH in query, or DTO projection.

Other lazy loading examples:

  • Spring @Lazy bean: Bean created only when first injected/used
  • Frontend: React lazy() / dynamic imports — component loaded on route visit
  • Images: loading="lazy" — image fetched when near viewport

What Is a Controller?

Role in layered architecture (Spring MVC)

Spring Boot

A Controller is the entry point for HTTP requests. It maps URLs to Java methods, validates input, delegates business logic to the Service layer, and returns HTTP responses.

@RestController
@RequestMapping("/api/users")
public class UserController {

    private final UserService userService;

    public UserController(UserService userService) {
        this.userService = userService;
    }

    @GetMapping("/{id}")
    public ResponseEntity<UserDto> getUser(@PathVariable Long id) {
        return ResponseEntity.ok(userService.findById(id));
    }

    @PostMapping
    public ResponseEntity<UserDto> createUser(@Valid @RequestBody CreateUserRequest req) {
        UserDto created = userService.create(req);
        return ResponseEntity.status(HttpStatus.CREATED)
            .location(URI.create("/api/users/" + created.getId()))
            .body(created);
    }
}
Controller should NOT: contain business logic, direct DB queries, or transaction management. That's the Service/Repository layer's job. Controllers handle HTTP concerns only — status codes, headers, request mapping, validation.

Request flow: Client → DispatcherServlet → Controller → Service → Repository → DB → back up the chain → JSON response.

What Is Indexing in Database? How Does It Work?

Deep explanation

SQL

An index is a separate data structure (usually B-Tree or B+Tree) that stores sorted copies of selected column values with pointers to the actual table rows. It lets the DB find rows without scanning the entire table (full table scan).

-- Without index: full table scan O(n) — reads every row
SELECT * FROM users WHERE email = 'ash@example.com';

-- Create index
CREATE INDEX idx_users_email ON users(email);

-- Now DB uses B-Tree lookup O(log n) — much faster on large tables
Index typeUse case
Single-columnFilter/sort on one column (email, created_at)
CompositeMulti-column queries ((dept_id, salary))
UniqueEnforce uniqueness + fast lookup
CoveringIndex includes all columns in SELECT — no table lookup needed
Trade-offs: Indexes speed up SELECT but slow down INSERT/UPDATE/DELETE (index must be updated). Too many indexes = write penalty + extra storage. Index columns used in WHERE, JOIN, ORDER BY.

You Got a P1 Issue in Production — How Will You Handle It?

Incident response playbook

Production / SRE
1. Acknowledge & communicate

Join war room / bridge call. Acknowledge alert. Update status page. Notify stakeholders: "We're investigating elevated error rates on payment API."

2. Mitigate first, root-cause second

Stop the bleeding — rollback last deploy, scale up, disable feature flag, route traffic away, restart unhealthy pods. User impact reduction is priority.

3. Investigate

Check dashboards (error rate, latency, CPU, DB connections), recent deploys, logs (correlate by trace ID), dependency health. Compare with last known good state.

4. Fix & verify

Apply hotfix or rollback. Verify metrics return to normal. Monitor for 30–60 min for recurrence.

5. Post-incident

Write blameless postmortem: timeline, root cause, action items (monitoring gaps, tests, runbooks). Prevent recurrence.

Key phrase: "Mitigate first, diagnose second. A rollback in 5 minutes beats a perfect fix in 2 hours during a P1."

Jenkins Pipeline Breaks During Production Deploy — Blue-Green Handling

Blue-Green deployment explained + failure handling

DevOps / CI-CD

Blue-Green: Two identical environments. Blue = live production. Green = new version. Deploy to Green, test, switch traffic, keep Blue as instant rollback.

Phase 1: Blue (v1.0) ← 100% live traffic
Phase 2: Deploy v2.0 to Green, run smoke/integration tests
Phase 3: Switch load balancer: Green ← 100% traffic
Phase 4: Monitor. If OK → decommission Blue. If FAIL → switch back to Blue
Pipeline fails mid-deploy

"If Jenkins fails while deploying to Green, production is unaffected because Blue still serves traffic. I would NOT switch the load balancer."

If traffic already switched

"Immediately rollback: point load balancer back to Blue. This is a DNS/LB config change — takes seconds, not a redeploy."

Investigate

"Check Jenkins logs for the failed stage (build, test, deploy, smoke test). Fix the issue on a branch, re-run pipeline to Green, test again before switching."

Prevention

"Automated smoke tests on Green before traffic switch. Manual approval gate. Canary deployment as an alternative — route 5% traffic first."

Multi-Step Flow Fails at Cab Booking — How to Avoid Further Steps?

Saga pattern + orchestration

Microservices / Distributed Systems

User flow: Plan trip → Book cab → Book train → Book flight → Book hotel. If cab booking fails, don't proceed — and undo any prior steps.

Trip Planning Saga (Orchestrator pattern):

Step 1: Book cab     → FAILS
Step 2: Book train    → SKIPPED (don't invoke)
Step 3: Book flight   → SKIPPED
Step 4: Book hotel    → SKIPPED

If Step 1 had succeeded but Step 3 failed:
  → Compensating transactions: Cancel cab, Cancel train (rollback)
PatternHow
OrchestratorCentral service controls flow; on failure, stops and triggers compensations
ChoreographyEach service publishes events; next step listens; failure event triggers compensations
// Orchestrator pseudo-code
try {
    CabBooking cab = cabService.book(cabRequest);     // FAILS here
    TrainBooking train = trainService.book(trainReq);
    FlightBooking flight = flightService.book(flightReq);
    HotelBooking hotel = hotelService.book(hotelReq);
    return TripConfirmation.of(cab, train, flight, hotel);
} catch (BookingException ex) {
    saga.compensate(completedSteps);  // cancel any already-booked steps
    return TripConfirmation.failed(ex.getMessage());
}
Key concepts to mention
  • Saga: Sequence of local transactions with compensating actions
  • Fail fast: Don't call train/flight/hotel if cab failed
  • Compensation: Cancel cab if hotel booking fails later
  • Idempotency: Safe to retry any step without double-booking
  • Event-driven alternative: Publish CabBookingFailed event; downstream consumers don't proceed

Resilience Patterns — Timeouts, Circuit Breakers, Bulkheads, Idempotency Keys

Learn in this order: 1 Timeout → 2 Circuit Breaker → 3 Bulkhead → 4 Idempotency Key. Each builds on the previous.

Correct learning sequence:
  1. Timeout — fail fast, free threads (foundation)
  2. Circuit Breaker — stop calling a dead service (uses timeout failures as signal)
  3. Bulkhead — isolate pools so one failure doesn't sink everything
  4. Idempotency Key — safe retries when clients retry after timeout/error

Why resilience matters

Foundation

In production, failures are normal — networks drop, services restart, databases slow down, users double-click Submit. Without resilience patterns, one slow dependency can exhaust all your threads and crash the entire application.

Without resilience:
  Payment service slow (30s response)
    → Order service threads all blocked waiting
      → Order service stops responding
        → Entire platform down (cascade failure)

With resilience:
  Timeout at 3s → thread released
  Circuit breaker opens → fail fast, no more calls to dead service
  Bulkhead → payment pool exhausted, but catalog API still works
  Idempotency key → client retry doesn't double-charge
PatternOne-line purposeAnalogy
TimeoutDon't wait forever for a responseHanging up a phone call after 30 seconds of silence
Circuit BreakerStop calling a failing service temporarilyElectrical breaker — trips when overload detected
BulkheadIsolate resources so one failure doesn't sink the shipShip compartments — one flooded section doesn't sink all
Idempotency KeySame request twice = same result onceReceipt number — showing it again doesn't charge twice

1. Timeouts — fail fast, free resources

Timeout

A timeout sets a maximum wait time for a call (HTTP, DB, message queue). If the downstream doesn't respond in time, your service aborts and returns an error — instead of holding a thread for 30+ seconds.

Why threads matter: Tomcat/Jetty has a fixed thread pool (e.g. 200 threads). If 200 requests are all waiting on a dead payment service, request #201 waits in queue → your API appears down even though only one dependency failed.
Timeout typeTypical valueWhere to set
HTTP client → downstream API2–5 secondsRestTemplate, WebClient, Feign
Database query5–30 secondsHikariCP connection timeout, query timeout
Overall request (gateway)10–30 secondsAPI Gateway, load balancer

// What happens on timeout:
// 1. Thread is released immediately (not blocked 30s)
// 2. Exception thrown → caught by circuit breaker or retry logic
// 3. User gets 504 Gateway Timeout or 503 with Retry-After
Rule of thumb: Set client timeout shorter than gateway timeout. If gateway waits 30s but your service waits 60s, the gateway kills the request while your thread is still blocked.

2. Circuit Breaker — stop hammering a dead service

Circuit Breaker

A circuit breaker monitors failure rate to a downstream service. When failures exceed a threshold, it opens — all subsequent calls fail immediately without hitting the dead service. After a cooldown, it tries again in half-open state.

Circuit Breaker States:

  CLOSED (normal)
    │  All requests pass through to downstream
    │  Failures are counted in sliding window
    │
    ▼  failure rate > threshold (e.g. 50% of last 10 calls)
  OPEN (tripped)
    │  All requests fail IMMEDIATELY (fail fast)
    │  No calls to downstream — gives it time to recover
    │  waitDurationInOpenState: 30 seconds
    │
    ▼  after cooldown
  HALF-OPEN (testing)
    │  Allow 1–3 test requests through
    │
    ├── success → back to CLOSED
    └── failure → back to OPEN
// Resilience4j Circuit Breaker — Spring Boot
@Service
public class PaymentService {

    @CircuitBreaker(name = "paymentService", fallbackMethod = "chargeFallback")
    public PaymentResponse charge(PaymentRequest req) {
        return paymentClient.charge(req);  // throws on timeout/5xx
    }

    // Called automatically when circuit is OPEN or call fails
    public PaymentResponse chargeFallback(PaymentRequest req, Throwable t) {
        log.warn("Payment circuit open or failed, returning graceful error", t);
        return PaymentResponse.failed("Payment temporarily unavailable. Please retry.");
    }
}

// application.yml
resilience4j.circuitbreaker:
  instances:
    paymentService:
      slidingWindowSize: 10              # evaluate last 10 calls
      failureRateThreshold: 50           # open if 50%+ fail
      waitDurationInOpenState: 30s       # stay open 30s before half-open
      permittedNumberOfCallsInHalfOpenState: 3
      slowCallDurationThreshold: 3s        # slow calls count as failures
      slowCallRateThreshold: 80
StateCalls to downstream?Response to caller
CLOSEDYes — normal operationActual response or error from downstream
OPENNo — fail immediatelyFallback response or 503 Service Unavailable
HALF-OPENLimited test callsTest if service recovered
Timeout vs Circuit Breaker: Timeout protects a single request. Circuit breaker protects the system by stopping all calls during an outage. Use both together.

3. Bulkhead — isolate failures by compartment

Bulkhead

A bulkhead limits concurrent calls to a dependency using a separate thread pool or semaphore. If the payment service is slow and exhausts its pool, the inventory and notification services still have their own pools.

Without bulkhead (shared pool of 200 threads):
  Payment calls:   180 threads blocked (payment service down)
  Catalog calls:    20 threads left  ← catalog starts failing too!

With bulkhead (separate pools):
  Payment pool:    50 threads max → all 50 blocked → payment fails fast
  Catalog pool:    50 threads max → all 50 still available → catalog works fine
  Notification pool: 20 threads max → unaffected
// Resilience4j Bulkhead — limits concurrent calls
@Service
public class OrderService {

    @Bulkhead(name = "paymentBulkhead", type = Bulkhead.Type.SEMAPHORE)
    @CircuitBreaker(name = "paymentService")
    @TimeLimiter(name = "paymentService")
    public CompletableFuture<PaymentResponse> processPayment(PaymentRequest req) {
        return CompletableFuture.supplyAsync(() -> paymentClient.charge(req));
    }

    @Bulkhead(name = "inventoryBulkhead", type = Bulkhead.Type.SEMAPHORE)
    public CompletableFuture<Boolean> checkInventory(Long productId) {
        return CompletableFuture.supplyAsync(() -> inventoryClient.check(productId));
    }
}

// application.yml
resilience4j.bulkhead:
  instances:
    paymentBulkhead:
      maxConcurrentCalls: 25       # max 25 simultaneous payment calls
      maxWaitDuration: 500ms       # wait 500ms for slot, then reject
    inventoryBulkhead:
      maxConcurrentCalls: 50

// Thread-pool bulkhead (for blocking I/O)
resilience4j.thread-pool-bulkhead:

4. Idempotency Keys — safe retries without duplicate side effects

Idempotency

An idempotency key is a unique client-generated ID sent with a request. The server stores the key → response mapping. If the same key arrives again (retry), the server returns the cached response without re-executing the operation.

Why retries happen (and why idempotency is essential):
  1. User clicks Pay → request sent
  2. Payment succeeds on server
  3. Network drops before response reaches client
  4. Client retries (or user clicks Pay again)
  5. WITHOUT idempotency key → customer charged TWICE
  6. WITH idempotency key → server returns original receipt, no second charge
// Full idempotency flow
POST /api/payments
Headers:
  Idempotency-Key: pay-order-99281-20240630   ← client generates ONCE per action
Body:
  { "orderId": "ORD-99281", "amount": 99.99 }

Server flow:
  Step 1: Check Redis/DB for key "pay-order-99281-20240630"
  Step 2a: Key EXISTS  → return stored response (HTTP 200, same body)
  Step 2b: Key MISSING → acquire lock, process payment, store result, return 201

@Service
public class IdempotentPaymentService {

    public PaymentResponse charge(String idempotencyKey, PaymentRequest req) {
        String storeKey = "idem:" + idempotencyKey;

        // Already processed?
        String cached = redis.get(storeKey);
        if (cached != null) return deserialize(cached);

        // Prevent concurrent duplicate processing
        if (!redis.setIfAbsent(storeKey + ":lock", "1", Duration.ofSeconds(30))) {
            throw new ConflictException("Request in progress");
        }
        try {
            PaymentResponse result = paymentGateway.charge(req);
            redis.set(storeKey, serialize(result), Duration.ofHours(24));
            return result;
        } finally {
            redis.delete(storeKey + ":lock");
        }
    }
}
RuleDetail
Who generates the key?Client — UUID or deterministic (userId + action + timestamp)
How long to store?24–48 hours (cover all retry windows)
Same response on retry?Yes — same HTTP status + body (don't error on duplicate)
Which methods need it?POST mainly; PUT/DELETE are naturally idempotent
Backup safety netDB UNIQUE constraint on business key (orderId + userId)

Calling the same request multiple times produces the same result as calling it once — critical for payments, orders, and retries.

// Server logic
@PostMapping("/payments")
public ResponseEntity<PaymentResponse> pay(
        @RequestHeader("Idempotency-Key") String idempotencyKey,
        @RequestBody PaymentRequest request) {

    Optional<PaymentResponse> existing = idempotencyStore.find(idempotencyKey);
    if (existing.isPresent()) {
        return ResponseEntity.ok(existing.get());  // same response, no double charge
    }

    PaymentResponse response = paymentService.charge(request);
    idempotencyStore.save(idempotencyKey, response, Duration.ofHours(24));
    return ResponseEntity.status(HttpStatus.CREATED).body(response);
}

// Idempotency store schema
CREATE TABLE idempotency_keys (
    key          VARCHAR(64) PRIMARY KEY,
    response     JSONB NOT NULL,
    created_at   TIMESTAMP DEFAULT NOW()
);
Rules for production idempotency
  • Client generates key once per user action; reuses same key on retry
  • Server stores key → response mapping for 24–48 hours
  • Combine with DB UNIQUE constraint on business key (orderId + userId)
  • Return same HTTP status + body on duplicate (don't error on retry)
  • Stripe, PayPal, and most payment APIs require this pattern

How Do You Prevent Duplicate Data When Multiple Users Submit the Same Request?

Duplicates come from retries, double-clicks, and race conditions — not just "two users."

// Redis-based idempotency — same key within 24h returns cached response
@Service
public class OrderService {

    @Autowired private StringRedisTemplate redis;
    @Autowired private OrderRepository orderRepo;

    public OrderResponse createOrder(String idempotencyKey, CreateOrderRequest req) {
        String cacheKey = "idem:order:" + idempotencyKey;

        // 1. Check if we already processed this request
        String cached = redis.opsForValue().get(cacheKey);
        if (cached != null) {
            return objectMapper.readValue(cached, OrderResponse.class);
        }

        // 2. Acquire short lock to prevent concurrent duplicate processing
        Boolean acquired = redis.opsForValue()
            .setIfAbsent(cacheKey + ":lock", "1", Duration.ofSeconds(30));
        if (Boolean.FALSE.equals(acquired)) {
            throw new DuplicateRequestException("Request already in progress");
        }

        try {
            // 3. DB unique constraint on (user_id, client_request_id) as safety net
            Order order = orderRepo.save(new Order(req, idempotencyKey));
            OrderResponse response = OrderResponse.from(order);

            redis.opsForValue().set(cacheKey, toJson(response), Duration.ofHours(24));
            return response;
        } finally {
            redis.delete(cacheKey + ":lock");
        }
    }
}

How Do You Ensure Transactional Consistency Across Multiple Microservices?

You cannot use a single 2PC database transaction across services — use distributed patterns instead.

Outbox patternEventualMediumGuarantee event published after DB commit

// Saga Orchestrator — Order flow
Order Saga:
  1. Create order       (Order Service)     ✓
  2. Reserve inventory  (Inventory Service) ✓
  3. Charge payment     (Payment Service)   ✗ FAILS
  → Compensate:
     2b. Release inventory
     1b. Cancel order

// Outbox pattern — atomic DB write + event publish
@Transactional
public void createOrder(Order order) {
    orderRepo.save(order);

    // Same transaction — event stored in outbox table
    outboxRepo.save(new OutboxEvent("OrderCreated", order.getId(), toJson(order)));
}
// Separate poller reads outbox → publishes to Kafka → marks as sent
// Guarantees: if order exists in DB, event WILL be published (eventually)
Interview answer

"Cross-service strong ACID isn't practical in microservices. I use Sagas for multi-step flows with compensating transactions — if payment fails, release inventory and cancel the order. For reliable event delivery I use the Outbox pattern: write business data and outbox event in the same local transaction, then a poller publishes to Kafka. Each service owns its data and uses eventual consistency with clear compensation paths."

// Spring Security — JWT + role-based authorization
@Configuration
@EnableMethodSecurity
public class SecurityConfig {

    @Bean
    SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        return http
            .csrf(csrf -> csrf.disable())          // disable only for stateless JWT APIs
            .sessionManagement(s -> s.sessionCreationPolicy(STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**").permitAll()
                .requestMatchers("/api/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated())
            .oauth2ResourceServer(oauth -> oauth.jwt(Customizer.withDefaults()))
            .build();
    }
}

@RestController
@RequestMapping("/api/admin/users")
public class AdminUserController {

    @DeleteMapping("/{id}")
    @PreAuthorize("hasRole('ADMIN')")
    public ResponseEntity<Void> deleteUser(@PathVariable Long id) {
        userService.delete(id);
        return ResponseEntity.noContent().build();
    }
}

What Is Your Deployment Strategy to Achieve Zero or Minimal Downtime?

# Kubernetes rolling update — zero downtime requirements
spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 0    # never drop below desired count
      maxSurge: 1          # spin up 1 new pod before killing old
  template:
    spec:
      containers:
      - name: my-app
        readinessProbe:    # only receive traffic when healthy
          httpGet:
            path: /actuator/health/readiness
            port: 8080
          initialDelaySeconds: 10
        lifecycle:
          preStop:         # graceful shutdown — finish in-flight requests
            exec:
              command: ["sh", "-c", "sleep 10"]

# Blue-Green flow
1. Blue (v1.0) ← 100% traffic
2. Deploy v2.0 to Green → smoke tests pass
3. Switch LB: Green ← 100%
4. Monitor 15 min → decommission Blue OR rollback instantly

# Database — expand-contract pattern (avoid breaking deploys)
Phase 1 (expand):  Add nullable column `email_v2` — deploy app reading both
Phase 2 (migrate): Backfill data
Phase 3 (contract): App writes only to `email_v2`; drop old column
Interview answer

"For zero downtime I use rolling updates in Kubernetes with readiness probes and graceful shutdown — maxUnavailable=0 ensures old pods stay up until new ones are healthy. For high-risk releases I prefer blue-green or canary: deploy to green, run smoke tests, shift 5% traffic, monitor error rate and p99, then full cutover. I use feature flags to decouple deployment from release. DB migrations follow expand-contract so schema changes never break the running version."
