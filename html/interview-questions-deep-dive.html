Sample 2-minute interview script

Talking Points
Step 1 — Architecture

"I'd design a horizontally scalable, stateless REST API. Traffic enters through a load balancer or API gateway for TLS, auth, and rate limiting. Stateless app instances run in an auto-scaling group."

Step 2 — Scale reads

"For millions of requests, most APIs are read-heavy. I'd add Redis cache-aside and optionally a CDN for cacheable GET endpoints to reduce DB load."

Step 3 — Database

"Primary for writes, read replicas for reads, with indexing and query optimization. Heavy work goes to an async queue."

Step 4 — Trade-off

"I'd avoid premature sharding. Scale vertically + replicas + cache first. Shard only when metrics prove we need it."

Follow-ups to expect: cache stampede (TTL jitter, request coalescing), hot keys (local cache, key splitting), strong consistency (read-your-writes via primary routing), write scaling (partition by tenant ID, CQRS).

API Gateway vs Load Balancer

Many engineers use these interchangeably — they aren't the same. In production you often need both.

The one-line difference

System Design
Load Balancer: Where should this request go? (which healthy instance)
API Gateway: What should happen to this request before it gets there? (policies, auth, routing)

Different responsibilities. Better together.

API Gateway — request & API policy layer

API Gateway

Acts as the single entry point for client requests at the application/API layer.

ResponsibilityWhat it does
Authentication & authorizationValidate JWT, API keys, OAuth tokens before traffic hits services
Rate limiting & throttlingProtect backends from abuse; return 429 with Retry-After
Request validationSchema validation, size limits, required headers
API versioningRoute /v1/users vs /v2/users to different backends
Protocol translationREST ↔ gRPC, HTTP ↔ WebSocket at the edge
RoutingSend request to the correct microservice (orders, payments, users)
ObservabilityCentral logging, metrics, tracing, request/response transformation

Focus: How a request should be processed before it reaches a service.

Examples: AWS API Gateway, Kong, Apigee, Spring Cloud Gateway, NGINX with auth plugins.

Load Balancer — traffic & availability layer

Load Balancer

Distributes incoming traffic across multiple instances of the same service for availability and scale.

ResponsibilityWhat it does
Traffic distributionRound-robin, least connections, IP hash across healthy nodes
Health checksStop sending traffic to failed/unhealthy instances
FailoverRoute around dead nodes automatically
SSL/TLS terminationDecrypt HTTPS at the edge; backends can use HTTP internally
High availabilitySpread load so no single server is overwhelmed

Focus: Which healthy instance should receive the request.

Examples: AWS ALB/NLB, NGINX, HAProxy, F5, Kubernetes Service + kube-proxy.

Typical microservices flow

Architecture 
Client
  → Load Balancer          (distributes across healthy gateway instances)
  → API Gateway            (auth, rate limit, validate, route by path/version)
  → Microservices          (User Service | Order Service | Payment Service)

Interview answer (30 seconds)

Talking Points
Script

"They're not interchangeable. A Load Balancer distributes traffic across multiple instances of the same service — it answers where the request should go based on health and load. An API Gateway is the application entry point — it handles auth, rate limiting, validation, versioning, and routes to the right microservice. It answers what should happen to the request before it reaches backend code. In production microservices, the typical flow is Client → Load Balancer → API Gateway → Services. You need both: LB for availability and scale at the infrastructure layer, Gateway for API policies at the application layer."

Common mistake in interviews: Saying "we use API Gateway for load balancing." Correct framing: the gateway routes to services; the load balancer distributes across instances. Some tools (e.g. NGINX, ALB) blur the line — acknowledge that, then explain the conceptual separation.

Find First Non-Repeating Character in String (Java 8)

Optimal approach — O(n) time, O(1) space (fixed alphabet)

DSA · Java 8

Two-pass with frequency array or HashMap. For ASCII/lowercase a–z, a int[26] or int[256] is O(1) space.

// Approach 1: LinkedHashMap — preserves insertion order, O(n) time
public static Character firstNonRepeating(String s) {
    if (s == null || s.isEmpty()) return null;

    Map<Character, Long> freq = s.chars()
        .mapToObj(c -> (char) c)
        .collect(Collectors.groupingBy(c -> c, LinkedHashMap::new, Collectors.counting()));

    return freq.entrySet().stream()
        .filter(e -> e.getValue() == 1)
        .map(Map.Entry::getKey)
        .findFirst()
        .orElse(null);
}

// Approach 2: Two-pass array — fastest for a-z, O(n) time, O(1) space
public static Character firstNonRepeatingFast(String s) {
    int[] count = new int[26];
    for (char c : s.toCharArray()) {
        count[c - 'a']++;
    }
    for (char c : s.toCharArray()) {
        if (count[c - 'a'] == 1) return c;
    }
    return null;
}

// Example: "swiss" → 'w'  |  "aabb" → null
Explain complexity
  • Time: O(n) — each character visited at most twice
  • Space: O(1) for fixed alphabet; O(k) for HashMap where k = unique chars
  • Why LinkedHashMap: HashMap doesn't guarantee order; we need first occurrence in string order

Find Duplicates in int Array (Minimal Time Complexity)

Approaches ranked by use case

DSA
ApproachTimeSpaceWhen to use
HashSetO(n)O(n)General case — find if duplicate exists or collect all dupes
Sorting + scanO(n log n)O(1)*When in-place sort allowed, memory constrained
Floyd's cycle (index as pointer)O(n)O(1)Array values in range [1..n], find one duplicate
Boolean array / bitsetO(n)O(n)Values in known small range [0..n]
// Find ALL duplicates — O(n) time, O(n) space
public static List<Integer> findDuplicates(int[] nums) {
    Set<Integer> seen = new HashSet<>();
    Set<Integer> duplicates = new LinkedHashSet<>(); // preserves order

    for (int num : nums) {
        if (!seen.add(num)) {  // add returns false if already present
            duplicates.add(num);
        }
    }
    return new ArrayList<>(duplicates);
}

// Java 8 stream variant — check if ANY duplicate exists
boolean hasDuplicate = Arrays.stream(nums).boxed()
    .collect(Collectors.groupingBy(i -> i, Collectors.counting()))
    .values().stream()
    .anyMatch(count -> count > 1);

// Find ONE duplicate when values are in [1..n] and array size n+1 — Floyd O(n), O(1)
public static int findDuplicateFloyd(int[] nums) {
    int slow = nums[0], fast = nums[0];
    do {
        slow = nums[slow];
        fast = nums[nums[fast]];
    } while (slow != fast);
    slow = nums[0];
    while (slow != fast) {
        slow = nums[slow];
        fast = nums[fast];
    }
    return slow;
}

How to Select Top 10 Rows from Database

Syntax by database + real use cases

SQL 
-- Standard SQL (PostgreSQL, SQL Server 2012+, Oracle 12c+)
SELECT * FROM employees
ORDER BY salary DESC
FETCH FIRST 10 ROWS ONLY;

-- MySQL / PostgreSQL shorthand
SELECT * FROM employees
ORDER BY salary DESC
LIMIT 10;

-- Oracle (legacy)
SELECT * FROM (
    SELECT e.*, ROWNUM rn FROM employees e ORDER BY salary DESC
) WHERE rn <= 10;

-- SQL Server (legacy)
SELECT TOP 10 * FROM employees ORDER BY salary DESC;

-- Top 10 PER department (window function — very common interview follow-up)
SELECT * FROM (
    SELECT *,
           ROW_NUMBER() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rn
    FROM employees
) ranked
WHERE rn <= 10;
Performance note: ORDER BY ... LIMIT 10 is fast only if there's an index supporting the sort column. Without an index, DB must sort the entire table. Mention this in interviews.

What Is Lazy Loading?

Concept + Hibernate/JPA example

JPA / Hibernate

Lazy loading defers loading of related data until it is actually accessed. Eager loading loads everything upfront in one query.

@Entity
public class Order {
    @Id
    private Long id;

    // LAZY (default for @OneToMany, @ManyToMany):
    // OrderItems are NOT loaded when you fetch Order
    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
    private List<OrderItem> items = new ArrayList<>();
}

// Usage — items loaded only when accessed (inside active session)
Order order = orderRepo.findById(1L);   // SELECT order only
order.getItems().size();                 // SELECT items NOW (lazy trigger)

// EAGER — loads order + items in one or multiple queries immediately
@ManyToOne(fetch = FetchType.EAGER)
private Customer customer;
LazyInitializationException: Accessing lazy collection outside a transaction/session throws this. Fix with @Transactional on service method, JOIN FETCH in query, or DTO projection.

Other lazy loading examples:

  • Spring @Lazy bean: Bean created only when first injected/used
  • Frontend: React lazy() / dynamic imports — component loaded on route visit
  • Images: loading="lazy" — image fetched when near viewport

What Is a Controller?

Role in layered architecture (Spring MVC)

Spring Boot

A Controller is the entry point for HTTP requests. It maps URLs to Java methods, validates input, delegates business logic to the Service layer, and returns HTTP responses.

@RestController
@RequestMapping("/api/users")
public class UserController {

    private final UserService userService;

    public UserController(UserService userService) {
        this.userService = userService;
    }

    @GetMapping("/{id}")
    public ResponseEntity<UserDto> getUser(@PathVariable Long id) {
        return ResponseEntity.ok(userService.findById(id));
    }

    @PostMapping
    public ResponseEntity<UserDto> createUser(@Valid @RequestBody CreateUserRequest req) {
        UserDto created = userService.create(req);
        return ResponseEntity.status(HttpStatus.CREATED)
            .location(URI.create("/api/users/" + created.getId()))
            .body(created);
    }
}
Controller should NOT: contain business logic, direct DB queries, or transaction management. That's the Service/Repository layer's job. Controllers handle HTTP concerns only — status codes, headers, request mapping, validation.

Request flow: Client → DispatcherServlet → Controller → Service → Repository → DB → back up the chain → JSON response.

What Is Indexing in Database? How Does It Work?

Deep explanation

SQL

An index is a separate data structure (usually B-Tree or B+Tree) that stores sorted copies of selected column values with pointers to the actual table rows. It lets the DB find rows without scanning the entire table (full table scan).

-- Without index: full table scan O(n) — reads every row
SELECT * FROM users WHERE email = 'ash@example.com';

-- Create index
CREATE INDEX idx_users_email ON users(email);

-- Now DB uses B-Tree lookup O(log n) — much faster on large tables
Index typeUse case
Single-columnFilter/sort on one column (email, created_at)
CompositeMulti-column queries ((dept_id, salary))
UniqueEnforce uniqueness + fast lookup
CoveringIndex includes all columns in SELECT — no table lookup needed
Trade-offs: Indexes speed up SELECT but slow down INSERT/UPDATE/DELETE (index must be updated). Too many indexes = write penalty + extra storage. Index columns used in WHERE, JOIN, ORDER BY.

You Got a P1 Issue in Production — How Will You Handle It?

Incident response playbook

Production / SRE
1. Acknowledge & communicate

Join war room / bridge call. Acknowledge alert. Update status page. Notify stakeholders: "We're investigating elevated error rates on payment API."

2. Mitigate first, root-cause second

Stop the bleeding — rollback last deploy, scale up, disable feature flag, route traffic away, restart unhealthy pods. User impact reduction is priority.

3. Investigate

Check dashboards (error rate, latency, CPU, DB connections), recent deploys, logs (correlate by trace ID), dependency health. Compare with last known good state.

4. Fix & verify

Apply hotfix or rollback. Verify metrics return to normal. Monitor for 30–60 min for recurrence.

5. Post-incident

Write blameless postmortem: timeline, root cause, action items (monitoring gaps, tests, runbooks). Prevent recurrence.

Key phrase: "Mitigate first, diagnose second. A rollback in 5 minutes beats a perfect fix in 2 hours during a P1."

Jenkins Pipeline Breaks During Production Deploy — Blue-Green Handling

Blue-Green deployment explained + failure handling

DevOps / CI-CD

Blue-Green: Two identical environments. Blue = live production. Green = new version. Deploy to Green, test, switch traffic, keep Blue as instant rollback.

Phase 1: Blue (v1.0) ← 100% live traffic
Phase 2: Deploy v2.0 to Green, run smoke/integration tests
Phase 3: Switch load balancer: Green ← 100% traffic
Phase 4: Monitor. If OK → decommission Blue. If FAIL → switch back to Blue
Pipeline fails mid-deploy

"If Jenkins fails while deploying to Green, production is unaffected because Blue still serves traffic. I would NOT switch the load balancer."

If traffic already switched

"Immediately rollback: point load balancer back to Blue. This is a DNS/LB config change — takes seconds, not a redeploy."

Investigate

"Check Jenkins logs for the failed stage (build, test, deploy, smoke test). Fix the issue on a branch, re-run pipeline to Green, test again before switching."

Prevention

"Automated smoke tests on Green before traffic switch. Manual approval gate. Canary deployment as an alternative — route 5% traffic first."

Multi-Step Flow Fails at Cab Booking — How to Avoid Further Steps?

Saga pattern + orchestration

Microservices / Distributed Systems

User flow: Plan trip → Book cab → Book train → Book flight → Book hotel. If cab booking fails, don't proceed — and undo any prior steps.

Trip Planning Saga (Orchestrator pattern):

Step 1: Book cab     → FAILS
Step 2: Book train    → SKIPPED (don't invoke)
Step 3: Book flight   → SKIPPED
Step 4: Book hotel    → SKIPPED

If Step 1 had succeeded but Step 3 failed:
  → Compensating transactions: Cancel cab, Cancel train (rollback)
PatternHow
OrchestratorCentral service controls flow; on failure, stops and triggers compensations
ChoreographyEach service publishes events; next step listens; failure event triggers compensations
// Orchestrator pseudo-code
try {
    CabBooking cab = cabService.book(cabRequest);     // FAILS here
    TrainBooking train = trainService.book(trainReq);
    FlightBooking flight = flightService.book(flightReq);
    HotelBooking hotel = hotelService.book(hotelReq);
    return TripConfirmation.of(cab, train, flight, hotel);
} catch (BookingException ex) {
    saga.compensate(completedSteps);  // cancel any already-booked steps
    return TripConfirmation.failed(ex.getMessage());
}
Key concepts to mention
  • Saga: Sequence of local transactions with compensating actions
  • Fail fast: Don't call train/flight/hotel if cab failed
  • Compensation: Cancel cab if hotel booking fails later
  • Idempotency: Safe to retry any step without double-booking
  • Event-driven alternative: Publish CabBookingFailed event; downstream consumers don't proceed

Step-by-step narrative

"When the user clicks Submit, the frontend validates input client-side, then sends an HTTP POST with JSON body and Authorization header. The request hits the load balancer, routes to our Spring Boot app. The DispatcherServlet maps it to the Controller method. Validation runs via @Valid. The Controller delegates to the Service layer, which contains business logic and starts a transaction. The Service calls the Repository, which persists to the database via JPA. On success, the Service returns a DTO, the Controller wraps it in ResponseEntity with 201 Created, Jackson serializes it to JSON, and the browser displays a confirmation."

Bonus points: Mention CORS preflight (OPTIONS) for cross-origin requests, HTTPS/TLS encryption in transit, and that the frontend should disable the Submit button during the request to prevent double-submit (or use idempotency key).
